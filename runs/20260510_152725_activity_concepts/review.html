ReasonWhat it meansWhy it blocks validity
Before/after evidenceThe product would need to compare a finished physical result with an earlier state or receive confirmation.Without an evidence policy, the activity could overclaim that it can assess drawings, builds, cleanup, or craft completion.
Caregiver setup and pacingThe activity requires adult setup, material timing, or a no-assessment physical-work fallback.The design needs clear caregiver responsibilities and pacing rules so the runtime does not leave the child waiting or judge unsupported work.
Cat3 material workflowThe child must use physical materials such as paper, blocks, craft objects, or handwriting.The current run contract supports Cat1/Cat5 packages; material setup, caregiver involvement, timing, and completion evidence need a product decision first.
Coloring or recoloring UIThe concept needs selectable regions, color application, or saved coloring state.Runtime beats would otherwise assume UI controls and state behavior that do not exist in the current package contract.
Motion safetyThe child is asked to move their body or follow pose/action prompts.Movement activities need age-safe constraints, caregiver gating, space checks, and prohibited movement rules before runtime dialogue is safe.
OCR or text handlingThe activity may need to read child writing, letters, words, or text content.Text recognition and reading-level behavior need explicit capability, privacy, and fallback decisions before runtime beats can rely on them.
Prebuilt asset displayThe activity depends on approved cards, artworks, pose images, or other displayed assets.The package needs known asset IDs, metadata, display rules, and fallbacks before it can claim what appears on screen.
Runtime image generationThe concept needs the product to generate child-facing images during the session.The package cannot promise runtime artwork until generation safety, latency, content policy, and fallback behavior are approved.
UI state or progress memoryThe activity needs persistent progress, resume/reset behavior, or interactive screen state.Designing rounds before the state model is defined risks creating flows the runtime cannot store, resume, or verify.
