# Runs

Run directories record provenance for each autonomous generation session and store the actual packages generated by that run.

Fresh assignment-generation runs create packages under `runs/<run_id>/activity_packages/<base_activity_id>/` with clean `activity_id` values, for example `runs/20260511_233559_activity_concepts/activity_packages/concept_scavenger_hunt_collect/`. Canonical/promoted packages may still live under `activities/<activity_id>/`, and checked-package enrichment may update the explicit `package_path=` or canonical package in place.

Use this layer to answer:

- Which `/goal` run produced this activity?
- Which assignments were pending at run start?
- Which assignments generated packages?
- Which existing packages were enriched to current standards?
- Which packages received independent reviewer-agent coverage?
- Which assignments blocked at the adaptation brief and constrained design preview?
- Where are the adaptation briefs, review notes, and human review dashboard for this run?

## Directory Shape

```text
runs/
└── <run_id>/
    ├── run_manifest.yaml
    ├── assignment_snapshot.md
    ├── generated_activity_ids.txt
    ├── activity_packages/
    │   └── <activity_id>/
    │       ├── spec.md
    │       ├── prod.md
    │       ├── tag_block.yaml
    │       ├── recap.template.yaml
    │       └── dashboard.template.yaml
    ├── adaptation_briefs/
    │   └── 001_<assignment_slug>.yaml
    ├── blocked_briefs/
    │   └── 002_<assignment_slug>.yaml
    ├── blocked_designs/
    │   └── 002_<assignment_slug>.md
    ├── generated_assets_pilot/
    │   └── <activity_id>/<asset_id>/
    │       ├── contact_sheet.prompt.md
    │       ├── contact_sheet.png
    │       └── asset.meta.yaml
    ├── integrated_assets/
    │   └── asset_bindings.yaml
    ├── activity_exports/
    │   ├── export_manifest.yaml
    │   └── <activity_id>.html
    ├── source_comparison/
    │   └── product_review_matrix.html
    ├── review_notes.md
    └── review.html
```

`run_id` format:

```text
YYYYMMDD_HHMMSS_<short_label>
```

Examples:

```text
20260509_113000_activity_concepts
20260509_150500_mapping_batch_04
```

Use local operator time for readability. Keep the timestamp in `run_manifest.yaml` as ISO 8601.

## File Roles

| File | Required? | Purpose |
|---|---:|---|
| `run_manifest.yaml` | Yes | Run-level index: inputs, status, enriched packages, audited no-op packages, generated packages, blocked briefs, constrained blocked design previews, checks, and timestamps. |
| `assignment_snapshot.md` | Yes | Copy of pending `assignments.md` rows at run start, preserving the source queue context. |
| `generated_activity_ids.txt` | Yes | One generated `activity_id` per line, in completion order. |
| `activity_packages/` | Yes for fresh generated packages | Five-file packages generated by this run. Directory names and `tag_block.yaml` `activity_id` values use clean base IDs; the run path distinguishes reruns. |
| `adaptation_briefs/` | Yes when Phase 0 runs | One YAML file per generated concept-led or underspecified assignment. |
| `blocked_briefs/` | Yes when blocked | One YAML file per blocked assignment. |
| `blocked_designs/` | Yes when blocked | One Markdown constrained design preview per blocked assignment, with detailed proposed runtime steps and inline `BLOCKED ELEMENT` comments. |
| `generated_assets_pilot/` | Optional post-run asset pilot | Review contact-sheet prompts, generated contact sheets, and metadata for selected prebuilt/display asset dependencies. These files do not change package contracts. |
| `integrated_assets/` | Optional when pilot assets are integrated | Run-local bindings between package asset IDs and generated asset files, generated by `scripts/integrate_generated_assets.py`. |
| `activity_exports/` | Optional when integrated activities are exported | One self-contained standalone reviewer packet per integrated activity plus `export_manifest.yaml`, generated by `scripts/export_activity_html.py`. Each packet embeds the integrated contact sheet and the review-only mechanism storyboard when available. |
| `source_comparison/` | Optional product review artifact | Product-facing source fidelity matrix generated by `scripts/generate_source_comparison_review.py`, comparing original workbook rows with generated packages and selective visual examples. |
| `review_notes.md` | Required for generated, enriched, or audited packages | Independent reviewer-agent coverage, PASS/FAIL/N/A evidence, repair notes, no-op audit decisions, re-review outcomes, and residual risks. |
| `review.html` | Required at run completion | Static human review dashboard generated by `scripts/generate_run_review.py` and governed by [`../review_dashboard.md`](../review_dashboard.md), with concise clickable cards for generated, enriched, and audited no-op packages, full detail dialogs, grouped tag colors, sortable/filterable package and blocked views, asset usage timelines for image/display dependencies, 10-dimension criteria and package scorecard results, resolved contract items, extensibility overview, blocked-preview scorecards, constrained preview links, reviewer coverage, checks, classified blocked-reason badges, colored inline blocked/resolved marker chips, and a reason guide. |

## Rules

- Create the run directory before processing the first assignment.
- Run review and asset/export scripts with a `python3` environment that has PyYAML available. If the local default `python3` lacks PyYAML, activate the repo/operator Python environment first, such as `env PATH="$(pyenv root)/versions/<version>/bin:$PATH" python3 ...`.
- Audit and, when needed, enrich checked-row packages before processing unchecked assignments. Use `package_path=` when present; otherwise use an existing canonical `activities/<activity_id>/` package.
- Spawn separate reviewer agents for package-quality review of generated packages and scoped enrichment/audit packages; partition package directories across reviewers when reviewing multiple packages in parallel.
- Keep fresh generated packages in `runs/<run_id>/activity_packages/<base_activity_id>/`. Do not write them under `activities/` unless explicitly promoting a package to the canonical package set.
- For fresh generation, treat assignment `activity_id=` as a base slug, strip an existing trailing `_rYYYYMMDD_HHMMSS` suffix, and keep the clean base slug as both the directory name and `tag_block.yaml` `activity_id`.
- Record changed enrichment-only package maintenance in `enriched_activities`; record no-op audit passes in `audited_activities`; do not append `results.tsv` for either path.
- Record reviewer-agent findings, direct repairs, author repairs, re-review outcomes, and residual concerns in `review_notes.md`.
- Record blocked assignments in `blocked_briefs/`, `blocked_designs/`, and `blocked_assignments`, leave their assignment rows unchecked, and continue to later unchecked rows.
- Blocked design previews still show the proposed detailed child experience, but every unsupported element must be annotated inline with `BLOCKED ELEMENT: <reason> -- <decision needed>`. One missing decision may appear as several inline markers when it affects several beats. They are not valid runtime packages and must not be copied into `activity_packages/` or `activities/`, logged in `results.tsv`, or checked off until the constraint is resolved and the standard package gates pass.
- If `product_contract_override=minimum_unblock_allowed` is active, generate formerly blocked minimum-to-unblock rows as normal packages, record former blockers in `resolved_blockers` / `resolved_blocker_types`, annotate affected `prod.md` beats with `RESOLVED BLOCKER`, and show those callouts in `review.html` without counting them as blocked assignments.
- For every prebuilt, displayed, or runtime-generated image dependency, record `asset_usage` in `run_manifest.yaml`, include `spec.md` `## Asset Usage Timeline`, and ensure `review.html` shows when and where the asset is generated/loaded, displayed, used, persisted or hidden, and replaced by fallback.
- Generated concept-led and parameterized packages should record `extensibility_summary` and `extensibility_notes` so `review.html` can show whether the activity can be retargeted to another entity, shared feature, matched property, or approved asset set.
- Update `run_manifest.yaml` after each enriched package, audited no-op package, generated package, or blocked assignment.
- At the end of the run, generate `review.html` with `python3 scripts/generate_run_review.py runs/<run_id>`, validate it with `python3 scripts/generate_run_review.py --validate runs/<run_id>`, confirm it satisfies `review_dashboard.md`, and record `outputs.review_dashboard`.
- When generated prebuilt asset files are integrated, generate `integrated_assets/asset_bindings.yaml` with `python3 scripts/integrate_generated_assets.py runs/<run_id>`, validate it with `python3 scripts/integrate_generated_assets.py --validate runs/<run_id>`, and record `outputs.integrated_asset_bindings`.
- When integrated activities are exported, generate standalone reviewer packets in `activity_exports/*.html` with `python3 scripts/export_activity_html.py runs/<run_id>`, validate them with `python3 scripts/export_activity_html.py --validate runs/<run_id>`, and record `outputs.activity_html_exports` plus `outputs.activity_html_export_manifest`. The packets should embed the integrated contact sheet and the review-only mechanism storyboard, with clear copy that the storyboard is not runtime UI.
- When product review needs workbook-to-generated comparison, generate `source_comparison/product_review_matrix.html` with `python3 scripts/generate_source_comparison_review.py runs/<run_id> --workbook <source.xlsx>`, validate it with `python3 scripts/generate_source_comparison_review.py runs/<run_id> --workbook <source.xlsx> --validate`, and record `outputs.source_comparison_review`. The page should show every source row, concise fidelity status, review questions, and only representative visual examples.
- Treat `generated_assets_pilot/`, `integrated_assets/`, and `activity_exports/` as derived run artifacts. Fix source manifests, package files, or generator scripts and regenerate; do not hand-edit derived HTML as source of truth.
- Treat `source_comparison/` as a derived run artifact. Fix source mappings, adaptation briefs, package files, export manifests, or the generator script and regenerate; do not hand-edit the review matrix HTML as source of truth.
- Commit `runs/<run_id>/` together with generated activity packages and queue/log updates for the run.
