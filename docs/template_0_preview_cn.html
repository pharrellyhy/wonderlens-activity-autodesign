§ 01 / 总览

三条诉求，
三处结构性答复。

内部讨论里提到的每一条诉求，都在 Template 0 里对应一个专门的位置。目的是把大家关心的事情直接写进 template 这一层——让每个设计都不可能绕过它们发出去。

01
"标签体系要够明确——要能对接上游的 app、儿童端的'小回顾'、家长端的'成长路径'。"

让 tag block 变成必填项

每个活动顶部都必须带一个 tag block，里面包含 entity、category、attributes、Key Concepts、Related Concepts、ATL——和内部讨论里提到的层级完全对齐。同一份 tag block 既喂上游的游戏选配管线，也喂下游的儿童端和家长端。

→ 参见 § 04 Tag Block · § 05 上下游对接
02
"进阶活动怎么设计——同样讲颜色，怎么一步一步稍微难一点？"

一套共用的 progression 轴

活动在 tag block 里声明 topic_axis 和 1–3 级难度。七条轴直接从项目里已经在用的 7 个 IB Key Concept 里长出来——思考透镜和进阶阶梯共用同一套名字，没有新东西要记。

→ 参见 § 07 进阶轴
03
"同一个游戏按交互去适配年龄，还是不同年龄各自做一套活动？"

相邻 tier 内部适配，跨度大就拆

一份设计先锚定一个主 tier，需要时可以向相邻 tier 延伸一步，靠五个明确的开关（AI 角色、语言、任务深度、概念聚焦、看护者角色）来调。跨两个 tier 就不是一份设计能扛的，应该拆成同一条 progression 轴上的两个相似活动。

→ 参见 § 06 Tier 适配
§ 02 / 主干

五节主干，
不可动摇。

所有 category——Cat1、Cat5，以及后面还没做的四个——都沿着同一条 5 节情感弧线。不同 category 之间变的是每一节的互动形式（纯对话、拍照收集、实地探索），节拍本身不变。

6 个 experience pillar —— 每个 pillar 决定 Beat 2–4 的情感走向
Mystery "我想明白了！"
Creation "这是我做的！"
Performance "他们好喜欢！"
Discovery "我猜对了吗？！"
Adventure "看我走了多远！"
Nurture "我帮到了！"
Beat 01

情感过渡

从照片出发，给一个情绪或想象上的回应。不要知识测验。

Beat 02

立角色 · 立框架

分配 role title 和 metaphor，用 pillar 专属的语言把任务说出来。

Beat 03

核心循环

N 轮 pillar 专属玩法。{escalation_axis} 必须沿着声明的 progression 轴往上走。

Beat 04

高光时刻

pillar 专属的汇总，情绪顶点。首次使用者也能从这一节认出是哪个 pillar。

Beat 05

庆祝 + 点名概念

把孩子刚刚达成的 Key Concepts 和 Related Concepts 作为称赞说出来——不是考试。

Cat1 和 Cat5 都只是薄薄一层 overlay，只需要覆盖 8 个字段：每节的互动形式、轮数、相机使用、场景、步数、核心玩法、anchor 维度优先级。 其他所有东西——pillar overlay、创作变量、entity 适配清单——都从 Template 0 继承。
§ 03 / 完整骨架

Template 0,
一整份文件。

§01 和 §02 把各个部件分开讲,这一节把它们拼回一整份文件——这就是 Template 0 在 templates.md 里实际的样子。Cat1、Cat5 以及未来的每一个 category 都只需要 extends: template_0 再覆盖 8 个字段,其余全部从这里继承。后面几节会逐块放大看。

§ 04 / Tag Block

每个活动，
都从一次对接开始。

这段 YAML 是每个设计最顶部必备的 block——也是游戏设计和上游 app、儿童端小回顾、家长端成长路径之间的契约。 高亮的字段是新增的，或者刚从可选变成必填。

activity_id: ladybug_cat1_mystery_t1
        entity: ladybug
        entity_type: animal
        category: cat1
        pillar: mystery
        style: mystery_lens
        tier: T1
        tier_variants: [T0] # 向相邻 tier 弹性延伸

        tags:                                 # 按下游面拆成两组
        # — 儿童端（直观的成果展示）—
        attributes: [red, black, spots, round]
        related_concepts: [Pattern, Camouflage] # 2–4 个孩子能认得的学科标签
        highlight_moment: "你发现了瓢虫背上的警告色" # 新增 —— 运行时生成
        # — 家长端（IB + 跨学科框架）—
        key_concepts: [Form, Change] # 从 7 个 IB Key Concept 里挑 1–2 个
        atl_skills: [critical_thinking, observation] # IB ATL
        transdisciplinary_theme: How the World Works
        subject_tags: [science, biology] # 新增 —— 跨学科分类

        kud:
        know: [瓢虫有 6 条腿, 不同种类斑点不同]
        understand: [图案可以帮助动物生存]
        do: [仔细观察, 描述所见]

        progression: # 新增 —— 回应诉求 #2
        topic_axis: connection
        difficulty_level: 2
        next_step_hint: "升到同一条轴的 L3，或跳到 causation L1"
        reward_hook: "connection_L2 徽章"

        caregiver_role: [scaffold, co-explorer] # 累加列表；默认值由 tier 决定

每个字段的用处——按服务的下游面拆开

下游用户分成两类。儿童端的活动回顾需要直观、看得见的"我做了什么"；家长端的成长路径需要 IB 框架 + 跨学科分类，把活动放进课程坐标。有些字段两边都用，大多数只服务其中一边。

儿童端 · 直观的成果展示
  • role_title孩子获得的角色头衔。动作感强、贴合他们实际做的事。
  • pillar情绪模式（Mystery、Creation 等）——变成任务的大框架。
  • related_concepts2–4 个孩子能认出的学科标签（图案、伪装…），作为徽章展示。
  • highlight_moment新增。运行时生成的一句话："你发现了 / 你想出了 / 你照顾了…"。是一份具体的奖杯，不是总结。
  • attributes游戏里用到的可观察属性——回放孩子当下注意到的东西。
家长端 · IB + 跨学科框架
  • key_concepts从 7 个 IB Key Concept 里挑 1–2 个——家长追踪"思维深度"的主轴。和 related_concepts 属于两套独立的分类。
  • atl_skills练到的 2–3 个 IB ATL 能力。家长可以跨活动看元能力的累积。
  • transdisciplinary_theme6 个 IB 跨学科主题中的哪一个。
  • subject_tags新增。跨学科分类（science / language / art / math / social-emotional）。家长可以按这个筛选，或和学校科目做对照——与儿童端有意错开。
  • progression活动瞄准的 axis + 难度等级，以及下一步建议。驱动成长路径里的阶梯视图。
  • tier · tier_variants清晰的审计线索——活动是按哪个年龄档调过的。
caregiver_role 是列表而不是单值——T0 只有 [scaffold]；T1 加上 co-explorer；T2 三者都在，但权重依次递减。目前只是软引导，即使看护者没到场，系统也不会拒绝开始。
§ 05 / 上下游对接

Tag Block，
是一份契约。

§04 里的字段都不是装饰。每一个都对接一个真实存在的上游或下游系统。有些系统今天已经有了（playbook 里描述的游戏选配管线），有些还没有——Template 0 就是让它们可以被具体设计出来的前提。本节沿两个方向各走一遍。

输入 孩子给物体拍照
→
识别 识别出 entity 和视觉属性
→
§04 Tag Block entity · category · attributes · tier · concepts · progression
→
选配 Tier A / B / P / 纯对话
→
运行时 活动开始

从照片到活动

孩子拍下一个物体后，游戏选配管线（参见 playbook §7）决定适配哪个活动。读取 tag block，按 Tier A > Tier B > Tier P > 纯对话的优先级排序。tag block 里每一个字段都能在这条管线里找到自己的出口。

Tier A · 10/10

Entity 金标准

该 entity 已经有一个专门设计的金标准游戏。管线直接按 entity + category 精确匹配。今天库里有 12 个（playbook §3A）。

读取的 tag block 字段
  • entity + category → 精确查找
  • tier → 语言 + 任务深度
  • pillar + style → 游戏玩法
  • key_concepts → 确认 KUD 对齐
Tier B · 8/10

Constellation 适配

没有直接的金标准，但有一个语义相近的 entity 有（比如小猫 → 狮子，秋千 → 游乐场）。运行时适配，不用新设计——只改 constellation_map.yaml。

读取的 tag block 字段
  • entity + entity_type → 分类 / 角色匹配
  • category → 必须一致（室内不能桥接户外）
  • attributes → 共享的 affordance
  • constellation 得分至少 0.7
Tier P · 7–8/10

Property-bridge template

既没有金标准也没有 constellation 匹配——但检测到的视觉属性可以触发一个参数化模板。今天库里有 18 个（playbook §3B）。一个模板可以服务成百上千个 entity。

读取的 tag block 字段
  • attributes → 最关键的匹配字段
  • category → 决定用哪套模板（Cat1 有 6 个 / Cat5 有 12 个）
  • tier → 模板内的语言弹性
  • 属性必须能从照片里看得出来
Template 0 的意义在这里。 把 attributes、entity_type、tier 三个字段提到 header 的一级位置，意味着每一个按 Template 0 画出来的金标准都可以被选配管线直接发现。playbook 描述了这条管线，而 tag block 让每一份新设计当天就能接进去——不用以后一遍一遍回头补元数据。

Tag block 喂的两个面

这两个面的完整设计各自成文。Template 0 负责提供它们共同依赖的契约。

§ 06 / Tier 适配 · 诉求 #3

五个开关，
逐 tier 调节。

建议：邻近 tier 之间做内部适配，跨度大就拆。一份设计先锁定一个主 tier，必要时通过五个明确的开关向相邻 tier 延伸一步。T0 ↔ T2 直接跨两级的认知差距太大——那种情况应该拆成同一条 progression 轴上的两个相似活动，而不是让一份设计硬撑。

AI 角色
热情的游戏发起者 + 情绪共鸣者
温暖、拟人化，会和孩子一起惊叹
语言
拟声词、叠词、3–5 词短句，语速慢、语调夸张
"汪汪！红色！大大的红瓢虫！"
任务深度
单步指令，即时反馈
"和我一起说——红！"
概念聚焦
具体属性——颜色、形状、质感——以及基本功能
"圆的。红的。会爬。"
看护者角色
陪玩者（唯一角色）
主动和孩子一起玩——跟读 AI 的话，和孩子一起把注意力保持在当下
AI 角色
好奇的共同探索者 + 思维启发者
和孩子一起纳闷，不下指令
语言
自然语速，完整句 + 简单复合句，开放式的"为什么 / 怎么样"
"你觉得瓢虫为什么身上有斑点？"
任务深度
2–3 步的序列任务，支持同伴或亲子的简单分工
"先找 3 个红色的东西，再告诉我哪个最大。"
概念聚焦
因果、变化、联系——从具体经验搭桥到抽象概念
"花为什么到晚上会合起来？"
看护者角色
陪玩者 + 共同探索者（权重大致各半）
孩子卡住时切回陪玩，其余时间分享自己看到的
AI 角色
项目顾问 + 资源协调者
把孩子当成小研究员对待
语言
接近成人的对话，复杂句式，用"首先……然后……"、"如果……那么……"之类的逻辑词
"我们先预测，再验证；如果结果出乎意料，就去查原因。"
任务深度
多步任务，可选的项目式分支，计划 → 实验 → 记录结果
"规划路线、收集证据，然后把你的发现讲一遍。"

§ 07 / 进阶轴 · 诉求 #2

七条轴，
来自七个 Key Concept。

七条进阶轴直接来自项目里已经在用的 7 个 IB Key Concept（参见 playbook §2 和 program.md §1.4）。 每一条轴就是孩子沿着一个 concept 往前走的轨迹。思考透镜和进阶阶梯共用同一套 7 个名字——没有新东西要记。

推导逻辑。 Form / Function / Causation / Change / Connection / Perspective / Responsibility 这七个本来就是每个活动挑"思考透镜"时用的 Key Concept。把它们同时当作进阶轴，意味着一个 T1 上的 Form-L2 设计和一个 T0 上的 Form-L1 设计会落在同一条阶梯上——孩子的成长故事读起来是沿着一个 concept 往前走，而不是从一个游戏毕业到下一个游戏。 颜色、形状、质感、大小都是 Form 的子例，只填在某一级台阶上，它们本身不是轴。
L1 / L2 / L3 在每一条轴上的共同含义
L1 识别 看见、指出或说出这个概念的一个实例。
L2 扩展 多找几个例子、多说几个部件、多走几步，或者换一个视角。
L3 推理 解释为什么、推演"如果……会怎样"，或说出把整组东西连起来的规则。
每条轴里具体的说法不同——Form 的 L1 是"说出一个主要属性"，Connection 的 L1 是"匹配一个相同属性"，Responsibility 的 L1 是"注意到一个需求"——但底层的认知动作是一样的：L1 第一次观察 → L2 扩展 → L3 解释。L1 / L2 / L3 是一条认知深度轴，和 T0 / T1 / T2 正交——后者只是根据年龄调整语言风格。一个 4 岁和一个 7 岁的孩子都可以处在 Form-L2 上，级数相同，只是说话方式不同。

Form — 属性描述

从说出一个可观察的属性，到讲清楚这个属性为什么重要。颜色、形状、质感、大小、图案都作为同一级台阶的不同子例出现。

L1
说出一个主要属性

孩子挑出一个可观察的属性并说出来——颜色、形状、大小、质感、图案、数量都可以。

"瓢虫是红的。" / "它是圆的。" / "它身上有斑点。"
L2
说出多个属性并指出位置

孩子同时说出 2–3 个属性，并且把每个属性落到 entity 的某个部位上。

"红色的身子，黑色的斑点，黑黑的小腿。"
L3
解释这个属性为什么存在

孩子对属性的目的做出解释——警告、伪装、成熟、信号、保护等。

"红色是在告诉小鸟'别吃我'。"

Function — 部件与功能

从指出部件，到预测某个部件如果少了会发生什么。经典的 part–whole 轨迹。

L1
指出并命名部件

孩子把 entity 可见的部件一一叫出名字，但不解释它们做什么。

"轮子。车门。车窗。车顶。"
L2
解释每个部件的作用

孩子把每一个部件和它的功能对上。

"轮子负责跑，车门是让人上车用的，车窗可以看外面。"
L3
预测少一个部件会怎样

孩子对缺失或损坏的部件做反事实推理。

"没有轮子？它就跑不动了。没有门？谁都上不来。"

Causation — 因与果

从描述发生了什么，到解释背后的机制。

L1
描述发生了什么

孩子如实描述观察到的现象，不做原因推断。

"花合起来了。"
L2
预测接下来会发生什么

孩子从当前状态出发，预测下一步可能发生的事。

"我把它搬到太阳底下，它应该就会再打开。"
L3
解释背后的机制

孩子给出一个因果故事来说明为什么会这样。

"花瓣要靠阳光才能撑开，晚上合起来是为了省力气。"

Change — 顺序与演变

从给两个时刻排序，到讲出一条有前因后果的完整演变轨迹。

L1
给两个时刻排顺序

孩子把两个状态按时间先后放对位置。

"先是绿香蕉，后是黄香蕉。"
L2
补上"之前 / 之后"

孩子补上缺失的第三个时刻，把序列补完整。

"之前是绿的，现在是黄的，之后会变成棕色、软软的。"
L3
讲出一条完整的演变故事

孩子把每一步之间的原因也讲出来，串成一条有因果的故事。

"种子喝了雨水，长出茎，长出叶子，开出花，最后又掉出新的种子。"

Connection — 图案与归类

从匹配一个相同属性，到说出整组东西共享的规则。Pattern recognition 和 classification 都放在这一条轴上。

L1
匹配一个相同属性

孩子再找到一个和原来那个东西共享一个可见属性的东西。

"这片叶子也是绿色的，和苹果一样！"
L2
找出 3 个以上的同类

孩子把相同属性扩展成一个小合集，并把它们归到一起。

"三个带斑点的东西：瓢虫、小球、小奶牛。"
L3
说出连接整组东西的规则

孩子说出这一组东西共享的属性或作用，并能解释这种分组为什么有意义。

"它们都有斑点。斑点可以帮它们躲开敌人。"

Perspective — 换位视角

从给 entity 安一种情绪，到让它去和第三者对话——共情也是一条阶梯。

L1
给 entity 安一种情绪

孩子把一种简单的情感安到 entity 身上。

"泰迪困了。"
L2
想象 entity 眼中的你

孩子反转视角，想象 entity 会怎么看自己、怎么感觉自己。

"泰迪觉得我又大又暖。"
L3
让 entity 对第三者说话

孩子替 entity 说话，而且不是在讲它自己，是在向另一个角色描述某件事。

"泰迪跟兔兔说：'我的小主人很温柔，她每晚都帮我盖被子。'"

Responsibility — 照料与影响

从注意到一个需求，到推理长期照料或不照料会发生什么。Nurture pillar 的天然归宿。

L1
注意到一个需求

孩子注意到 entity 有需要——脏了、干了、孤单、坏了。

"花蔫了。"
L2
提出一个具体的照料动作

孩子能说出一个具体的、能帮上忙的行动。

"我给它浇点水，再把它搬到窗边。"
L3
推理长期影响

孩子讲出持续照料会怎么样，以及不照料又会怎么样。

"每天给它浇水它就会长大。我忘了浇，它的叶子就会干掉、掉下来。"

升/降级 —— 什么时候在级数之间移动

评审里的两个子问题：(1) L1/L2/L3 够不够细？(2) 什么时候把孩子往上升、什么时候往下挪？答案是：三级就是承诺，同一级内部靠五个 tier 开关把任务深度拉开。级数之间的切换先用可量化的硬指标跑起来——先把能测的东西推上线，等有了足够的数据，再用一个按孩子个体调的节奏模型替换这些阈值。

活动内部
逐轮 · 同一次活动里的递进
↑ 升级 孩子在连续 2 轮以上不需要重复提示就能完成当前级，或者在当前级上自发做出了 L+1 的回答（如：只让他命名，他自己讲出了原因）。
· 保持 答得慢但没跑题。留在当前级，只换例子来保持新鲜感。
↓ 降级 孩子连续 2 次尝试都没能完成当前级（沉默超过 6 秒、跑题、复读提示）。降一级，仍在同一条轴上——绝不在活动中途弃轴。
活动之间
逐次 · 同一条轴上的长期进阶
↑ 升级 同一条轴上连续 3 次活动都以当前级完成——下一次该轴活动默认升到 +1 级。
· 保持 连续成功不到 3 次，或不同轴上表现不一。保留当前级，换 entity 和 pillar 增加多样性。
↓ 降级 孩子在同一条轴上连续 2 次都出现了活动内降级——下一次该轴活动默认降到 −1 级（最低保底 L1）。
不让孩子挫败 降级从不以"失败"的语气呈现。话术从"再试一次"切换到换一个切入点："我先告诉你我看到了什么"。孩子感受到的是多样性，不是退步。
同辈跳跃 如果孩子在一条轴上已经到 L3，下一次活动可以跳到一条相关轴的 L1（Form L3 → Connection L1），不必硬在别的轴上也塞 L3。既保持新鲜感，又不强推深度。
个性化接口 这些阈值是 tag block 上游配置里的 progression_policy 对象——不写死。V1 用上面的默认值；等每个孩子累计约 ≥ 200 次会话后，直接替换成按孩子个体的节奏模型，不需要改任何活动设计。

§ 09 / 决策记录

四个问题，四个决定。

这些原本都是评审前的待决问题。内部讨论后的回复现在已锁定，直接决定了 Template 0 的最终形状。

01
topic_axis 预设几条，从哪里推出来？
七条，对应 7 个 IB Key Concept——Form、Function、Causation、Change、Connection、Perspective、Responsibility。 上一稿挑的轴（比如 "color_naming"）太具体了，其实只是 Form 的一个子例。直接从 Key Concept 推导的好处是：思考透镜和进阶阶梯共用同一套 7 个名字，团队不用再记一套新词汇。
已定
02
tier_variants 写在 header 里，还是靠 runtime 推断？
写在 header 里。家长端看板和审计都需要显式信息。作者多花一点点成本换可追溯性。
已定
03
key_concepts 和 related_concepts 算同一套分类，还是两套？
两套独立的分类。 Key Concepts 是 IB 的"思考透镜"（通用的 7 个）。Related Concepts 是带学科标签、面向儿童的徽章。家长端看板可以直接把它们分两列展示。
已定
04
T0 的看护者角色是硬门槛还是软引导？
目前是软引导。 T0 默认 caregiver_role: scaffold，但系统不会因为没有看护者就拒绝开始。 等真实使用数据出来之后再决定要不要升级为硬门槛。
已定
