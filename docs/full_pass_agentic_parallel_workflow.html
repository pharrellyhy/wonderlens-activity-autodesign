Parallel full-pass workflow

Use this guide when several people work in the same repository on one full agentic pass. The rule is simple: one run branch, one run captain, activity-scoped text and asset work, and downstream truth checked separately by integration.

Choose a role See branch model PM review criteria Onboarding doc Agent prompts

Role Split

Everyone can inspect the full run, but each person edits a narrow area. This keeps parallel work reviewable and prevents package text, asset manifests, and status files from becoming merge hot spots.

Click role filters in the sidebar

Person A

Run Captain

A

Owns the canonical pipeline contract and integrates the run.

  • Freezes the canonical goal, program, run loop, templates, and assignments.
  • Creates the baseline run output and run branch.
  • Merges role PRs, reruns validators, and prepares handoff.

Person B

Text Reviewer

B

Owns whether assigned activities are playable and runtime-ready.

  • Reviews package text, metadata, and recap/dashboard templates.
  • Checks parameterization, step instructions, and branch quality.
  • Marks text status with concrete repair notes.

Person C

Asset Curator

C

Owns image completeness, reference fidelity, and round-screen fit.

  • Reviews asset manifests, package-local PNGs, and source metadata.
  • Generates illustrative sources with Codex imagegen or the Gemini provider.
  • Rejects fake reference-bound assets and mixed contact sheets.

You

Integration Owner

I

Owns downstream truth and PM-facing acceptance evidence.

  • Runs WonderLens AI and fullstack-demo import/runtime checks.
  • Separates package-owned failures from consumer-runtime failures.
  • Publishes PM review packet and final verdict.

Parallel Work Rule

  • One package can have one text owner and one asset owner at the same time.
  • Do not let two people edit the same package text or asset manifest concurrently.
  • Split shared status per activity instead of editing one global tracker from many branches.
  • Large runs use small PR batches, such as activities 01-05 and 06-10.

Merge Order Per Activity

  • Merge text/package PR first when the asset list depends on final runtime steps.
  • Merge asset PR after rebasing on latest text changes.
  • Run captain then updates validation evidence and dashboard output.
  • Integration owner runs downstream checks only after package and asset gates pass.

Gate Meanings

Pass Evidence exists and the owner accepts the risk.
Needs fix The owner identified a repair with clear file scope.
Blocked Product capability, missing source, or unsupported mechanic prevents acceptance.
Pending No reviewer has produced evidence yet.

Human Review

QA passing proves structure and runnable behavior. PM review needs a fast way to judge whether the activity feels good for a child and worth demonstrating.

PM-ready evidence

1. Package

What is it?

Title, category, mechanic, age band, entity mode, support level, and one-line child experience.

2. Visuals

What appears?

Icon, intro, rules, round scenes, item cards, and reference source thumbnails where relevant.

3. Playback

How does it play?

Intro -> child action -> AI response -> retry or next step -> celebration -> closing.

4. Quality

Why is it good?

Score clarity, actionability, runtime fidelity, visual correctness, recovery, and child fit.

5. Verdict

What happens next?

Ready to demo, needs polish, blocked by product capability, or not worth promoting.

PM Criterion Pass Looks Like Common Failure Owner
Goal clarity Child knows what to do after one prompt. AI gives theme copy but no concrete action. Text Reviewer
Actionability Each beat asks for one tap, choice, voice response, drawing confirmation, or photo. Multiple actions are bundled into one step. Text Reviewer
Runtime fidelity Converted prompts preserve package runtime instructions. Downstream runtime falls back to generic encouragement. Integration Owner
Visual correctness Assets are readable, style-consistent, and factually correct for reference cases. Constellation, art, or object reference is random or unverifiable. Asset Curator
Recovery behavior Wrong, unclear, or silent input gets a beat-specific retry. Every retry uses the same generic hint. Text Reviewer
Demo fit Activity is supported, honestly degraded, or gated as unsupported. Unsupported mechanic is hidden behind a Cat1/Cat5 scaffold. Run Captain

When a reviewer does not use Codex

Text/package review can be done manually from Markdown and YAML. Asset review can also be manual for source curation, prompt review, manifest completeness, and visual QA. Actual generated PNG creation can use a Codex-enabled operator or the Gemini provider documented in docs/image_generation_provider_setup.md. The provider writes only source PNGs; the deterministic builder still creates package-local runtime variants.

When to rerun the agentic pipeline

Rerun when the failure is systemic: repeated thin runtime instructions, repeated missing parameterization, repeated wrong asset granularity, or repeated unsupported mechanic misclassification. Patch packages directly when the failure is local to one activity.

What the live dashboard should add later

The live dashboard should read run_manifest.yaml, per-activity status YAML, package manifests, validation reports, and downstream reports. It should render the same gates shown here, then provide filters for blocked activities, missing assets, Cat5 runtime judgment, PM-ready activities, and owner-specific tasks.
