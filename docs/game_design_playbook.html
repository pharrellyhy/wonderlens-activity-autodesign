Project Status Overview

The WonderLens Game Design system is being built in phases. This overview tracks what has been completed, what is being worked on next, and what is planned for later. Each card links to the source files where the detailed implementation or spec lives.

DONE Spec and/or implementation complete — ready to use.   NEXT UP Design complete, implementation is the next priority.   PLANNED Designed but not yet started — deferred behind higher-priority work.

Infrastructure
The core reference files that define how games are structured — the style taxonomy (12 game styles organized under 6 emotional pillars), the templates designers fill in, the quality rubric, and the automated generation workflow. These are the "rules of the game" for building games.
docs/game_styles.md · templates.md · program.md · run.md
12 Entity Gold Standards
Hand-crafted exemplar games — one for each of the 12 game styles. Each is designed for a specific entity (e.g., lion, banana, teddy bear) and serves as the quality bar that all other games are measured against. These are the "best possible" versions of each game type.
designs/cat1/*_gold_*.md · designs/cat5/*_gold_*.md
18 Property-Bridge Templates
Reusable game templates that work for ANY entity based on a detected visual property (e.g., "this thing is round" → launch a shape-finding game). Unlike gold standards which are designed for one entity, each property-bridge template serves hundreds of entities. 12 outdoor + 6 indoor templates covering all styles.
designs/cat5/*_property_gold_*.md · designs/cat1/*_property_gold_*.md
Style Recommender Spec
The design for the system that decides WHICH game to play for a given entity. When a child photographs something, this system analyzes the conversation context, matches the entity to the best game style, and finds the most compatible existing game to reuse or adapt.
docs/plans/2026-04-01-style-recommender.md
Entity Bridging Spec
The design for handling entities that have NO pre-built game. Three fallback layers: (1) bridge to a similar entity's game through natural conversation, (2) detect a visual property and use a property-bridge template, (3) end with a rich conversation instead of a bad game.
docs/plans/2026-04-08-entity-bridging.md
10-Dimension Rubric
The quality checklist every game must pass before shipping. 10 dimensions covering technical limits, emotional hooks, edge cases, IB curriculum alignment, age-appropriate language, dialogue quality, screen design, entity grounding, genuine fun ("Game Feel"), and emotional identity ("Pillar Fidelity").
program.md (§Phase 3 rubric)
Implementation Prompts (Track C)
The actual LLM prompts that power the Style Recommender and Entity Bridging at runtime. These are the "instructions" the AI follows when deciding which game to play and how to bridge between entities. Currently designed but not yet written as prompt files.
prompts/stage1_conversation_analyzer.md · prompts/stage2_style_matcher.md · constellation_map.yaml · conversation_bridge.md §5
Legacy Design Rewrite (Phase 2)
~25 games created under the old 6-style system need to be rewritten into the new 12-style pillar system. Paused — the 12 gold standards + 18 property-bridge templates already provide a working game library for launch.
Constellation Map Population
Building a lookup table that maps common entities children photograph (cats, dogs, toys, household items) to existing gold standard games they can reuse. This is the cheapest way to expand game coverage — no new game design needed, just mapping.
Cleanup (Phase 3)
Housekeeping — archive old-format files, update handoff documentation, rebuild the concatenated design document for the full game library.

How to Choose a Pillar

Ask: "After 3 minutes of play, how should the child FEEL?"

If you want the child to feel… Choose The game should have…
Clever, like a detective Mystery Hidden information, progressive clues, "aha!" reveal
Proud of something they made Creation Open-ended building, snowballing modifications, "ta-da!" moment

Full file: designs/cat1/toy_robot_cat1_gold_spec.md (197 lines) — below is a visual walkthrough of the key sections.

Entity
Toy robot
Tier
T1 (ages 4–6)
Pillar

Child's Role
Chief Robot Inspector

How it works: AI secretly picks a detail on the child's own robot photo and feeds progressive clues — shape, function, location. The child looks at their photo and deduces which part. 3-4 rounds of escalating difficulty.

Step 1 — Emotional Hook
AI: (amazed) "Whoa, look at this robot! It has SO many cool parts — I see wheels, buttons, a pointy antenna... If your robot could talk, what do you think it would say?"
3 branches: Ideal → "Beep boop!" / Unexpected → "It has red eyes!" / Silence → wait 2s, whisper prompt
Screen: Robot photo centered with glowing pulse. Gear-shaped sparkles around edges.
Step 2 — Game Setup + Demo
AI: "I'm going to think of ONE secret part. I give clues, you guess! Watch — something round... helps your robot move... The WHEELS! Now let's play for REAL. Can you figure it out before my third clue?"
Screen: Robot photo left, "ROBOT INSPECTOR" magnifying glass right, demo animation, "Case 1" badge.
Round 1 — The Antenna (easy — build confidence)
AI: (mysterious) "Case one! It is thin and pointy — like a tiny stick poking up!"
Ideal: "The antenna!" → "You GOT it on the FIRST clue! It picks up signals — like a tiny ear listening to the sky!"
Wrong: "The arm!" → Clue 2: "It sticks UP from the very top of the head. Like a flag pole!"
Silence: → "Look at the very TIP TOP of your robot..."
Round 2 — The Button (introduces function)
AI: "Something small and round... and you can PUSH it with your finger!"
Ideal: "A button!" → "Buttons are like tiny commands!" · Wrong (eyes): → "You can push it IN, like a doorbell..."
Round 3 — The Joint (tricky — hidden part)
AI: (whisper) "This one is sneaky! Hidden where two pieces CONNECT... it helps your robot BEND."
Ideal: "The bendy part!" → "Without joints, your robot would be stiff like a stick!" · Names limb: → "Look WHERE the arm bends..."
Step 4 — Magic Moment
AI: (building excitement) "Chief Inspector... you inspected the antenna — the signal-catcher! The button — the commander! The sneaky joint — the bendy connector! Every single part — SOLVED! You are officially the CHIEF ROBOT INSPECTOR!"
Screen: Golden "CHIEF ROBOT INSPECTOR" badge. Solved part icons orbit with checkmarks. Confetti burst.
Step 5 — IB Closing
AI: "You noticed what each part LOOKS like — that's Form! You figured out what each part DOES — that's Function! You didn't just play with your robot. You INSPECTED it."

D10Pillar FidelityPASSMystery: clues → deduction → aha! Could NOT be re-labeled to another pillar.

3B. Property-Bridge Templates Tier P

What: A game parameterized by a detected visual property (e.g., "round," "red," "metal"), not a specific entity. Works for ANY entity with that property.

Quality: Good (7-8/10). Structured game with real mechanic, but not entity-tailored.

Current inventory: 18 property-bridge templates (12 Cat5 + 6 Cat1)

3C. Constellation-Adapted Games Tier B

What: An entity gold standard game reused for a semantically similar entity. No new game file needed — adaptation happens at runtime.

Example: Kitten → no gold standard → kitten is in lion's constellation → lion's game adapted: replace "roar" with "meow," "jungle" with "neighborhood."

When to expand: Add entries to constellation_map.yaml. No game design needed — just mapping.

4 How to Design an Entity Gold Standard DONE ›
Source: templates.md · entity_guidance.md · conversation_bridge.md · transform.md

Step-by-step

  1. Choose entity + category + tier + pillar + style — Entity from IB mind map, category based on where encountered, tier by age, pillar by desired emotional experience
  2. Read the entity mapping YAML — Note IB themes, primary key concepts (MUST use ≥1), tier_guidance, select 2-3 anchor dimensions
  3. Read the base template + pillar overlay from templates.md
  4. Study the gold standard for your style — Read both spec and prod
  5. Fill creative variables — Universal: {metaphor}, {role_title}, {escalation_axis}. Pillar-specific per templates.md. Must be FRESH.
  6. Write the spec — Step 1 (Hook), Step 2 (Setup+Demo), Step 3 (Core Loop 3-4 rounds), Step 4 (Magic Moment), Step 5 (IB Celebration). Each step: dialogue + 3 branches + screen.
  7. Self-evaluate against 10 dimensions — Fix failures, re-evaluate until ALL PASS
  8. Write the prod version — Bullet→table, condense rounds 2+, remove scorecard
  9. Save with correct naming — {entity}_cat{N}_gold_spec.md + _gold_prod.md

Spec File Format

# Activity Design: {Entity} + Category {N} ({Category Name})

            > Generated: {date} | {style info} | Agent: Activity Design Agent

            ---

            ## Activity: {Activity Name}

            ### A. Basic Info
            - **Activity Name**: ...
            - **Experience Pillar**: ...
            - **Game Style**: ...
            - **Design Version**: 1.0
            (bullet list format — see reference files)

            ### B. Activity Overview
            - ① Brief Description
            - ② Educational Purpose (KUD)
            - ③ Design Highlight
            - ④ Typical Scenario

            ### C. Interaction Flow — Detailed Design [Target Tier: {T}]
            (Steps 1-5, full dialogue, 3 response branches, screen descriptions)

            ## Self-Evaluation Scorecard
            (10 dimensions, ALL PASS)

            | Field | Value |
            |-------|-------|
            | Activity Name | ... |
            (table format, no scorecard, condensed rounds)

6 Quality Rubric: 10 Dimensions DONE ›
Source: program.md (§Phase 3 rubric)

Every game must pass ALL 10 dimensions.

# Dimension PASS when… FAIL when…
D1 V1 Technical Compliance No OCR, face detection, IMU, or before/after comparison Design depends on blocked capability
D2 Hook & Transition Step 1 opens with emotional resonance, grows naturally Step 1 tests knowledge or assigns task abruptly
D3 Edge Case Coverage Every step has 3 branches (ideal, unexpected, silence) Missing branches or unexpected responses dismissed
D4 IB Completeness 1-2 Key Concepts named. KUD defined. ATL skills identified. Concepts missing or don't match what child did
D5 Tier Appropriateness Language matches tier. T0: ≤5 words. T1: 5-8. T2: complex. Language too advanced or simple for tier
D6 Dialogue Specificity All AI lines are actual dialogue with tone markers Abstract descriptions like "AI guides…"
D7 Screen & UI Every step has specific screen description Vague or missing screen descriptions
D8 Entity Mapping Alignment (Mapping-informed only) Concepts from mapping Concepts not traceable to mapping
D9 Game Feel Genuine uncertainty + satisfying resolution + stakes Feels like structured Q&A with no surprise
D10 Pillar Fidelity Blind reader could identify pillar. Emotional arc matches. Could be re-labeled without feeling wrong

7 Game Selection Pipeline SPEC ›
Source: docs/plans/2026-04-01-style-recommender.md · docs/plans/2026-04-08-entity-bridging.md
Child photographs entity ↓ Entity in IB mind map? ──YES──→ Entity has gold standard game? │ ├── YES → Tier A (gold standard) │ └── NO → Style Recommender → Tier B (adapt) │ → or Tier P │ └── NO → Entity Bridging System ├── Constellation match (≥ 0.7)? │ └── YES → Bridge in conversation → Tier A/B │ ├── Property bridge available? │ └── YES → Tier P (property-bridge template) │ └── No match → Conversation-only (no game)

Entity Constellation Bridge Rules

  1. One bridge attempt per conversation — never retry if child ignores it
  2. Honor the original entity — bridge adds a connection, never dismisses
  3. Category must match — don't bridge indoor entity to outdoor game
  4. Minimum constellation score 0.7
  5. Game references BOTH entities — "Since your kitten is a cousin of a lion…"

Bridge Types (strongest → weakest)

  1. Same taxonomy (kitten → lion)
  2. Part-of (swing → playground)
  3. Same domain (tulip → sunflower)
  4. Shared affordance (toy car → bicycle)
  5. Same role (stuffed bunny → teddy bear)
  6. Companion (toothbrush → toothbrush holder)
  7. Shared habitat (pond frog → goldfish)
8 Automated Game Design Pipeline NEXT UP ›
Source: run.md · assignments.md · program.md · transform.md

Games can be generated automatically using an LLM agent. The agent reads entity mapping data, applies templates, generates both spec and prod files, self-evaluates, and commits — with no human intervention.

Setup (one-time)

File What the agent learns
program.md Constraints, output format, rubric, exemplars
templates.md Structural skeletons + pillar overlays
docs/game_styles.md 12 game styles taxonomy
entity_guidance.md How to read entity mapping YAMLs
conversation_bridge.md Warm/cold start bridge patterns
transform.md Spec → prod conversion rules
assignments.md Work queue (what to generate)

Assignment Queue Format

- [ ] goldfish + category 1, tier=T1, mapping=animals_goldfish,
            start=warm+cold, scene=child photographs their pet goldfish
Field Required Description
entity Yes The entity name
category Yes 1 (Cat1) or 5 (Cat5)
tier Yes T0, T1, or T2
pillar No Experience pillar (inferred if omitted)
style No Game style (inferred if omitted)
mapping No Entity mapping ID — enables mapping-informed design
start No warm+cold for dual bridge, or omit for cold-start only
scene No Trigger scene (invented if omitted)

The Design Loop

Step 1: Parse assignment → extract entity, category, tier, pillar, style, mapping ↓ Step 1.5: Load entity mapping YAML (if mapping= specified) → Extract IB themes, key concepts, tier_guidance → Select 2-3 anchor dimensions ↓ Step 2: Load base template + pillar overlay → Brainstorm creative variables grounded in mapping data ↓ Step 3: Generate full activity design (spec format) → 5-6 steps, full dialogue, 3 response branches, screen descriptions ↓ Step 4: Self-evaluate against 10 dimensions → Fix failures, re-evaluate until ALL PASS ↓ Step 5: Save spec → designs/{entity}_cat{N}_gold_spec.md ↓ Step 5.5: Transform spec → prod (apply transform.md rules) → Save prod → designs/{entity}_cat{N}_gold_prod.md ↓ Step 6: Log result → results.tsv ↓ Step 7: Mark assignment [x] complete ↓ Step 8: Git commit (spec + prod + results.tsv + assignments.md) ↓ Step 9: Next assignment → repeat

Mapping-Informed vs Non-Mapping

Aspect Mapping-informed Non-mapping
Entity data Full IB YAML (themes, concepts, tier_guidance) None — agent uses general knowledge
Bridge type Warm + cold start (Step 1a + 1b) Cold start only (Step 1b)
D8 evaluation Scored N/A
Quality Higher — grounded in researched IB data Good — but less IB-grounded

Batch Tips

  • Parallelize: Different pillars can run simultaneously on separate agents
  • Exemplar-first: Generate 1 gold standard per style before scaling
  • Mix tiers: Vary T0/T1/T2 for diversity
  • Diversify entities: Different types (animal, object, place) per style

12 Quick Reference Checklists ›

Before You Start

  • Read the gold standard for your target style (spec + prod)
  • Read the pillar overlay in templates.md
  • If mapping-informed: read the entity mapping YAML
  • Confirm entity + pillar pairing makes sense
  • Check existing inventory — does a similar game exist?

During Design

  • Step 1 opens with emotion, not knowledge testing
  • Every step has 3 response branches (ideal, unexpected, silence)
  • All AI dialogue has tone/emotion markers
  • Screen descriptions are specific (not "an animation plays")
  • Language matches the target tier
  • The magic moment is clear and emotionally distinct
  • Rounds escalate (don't repeat same difficulty)

After Design

  • Self-evaluation scorecard — ALL 10 dimensions PASS
  • D9 check: genuine uncertainty + satisfying resolution?
  • D10 check: blind reader could identify the pillar?
  • Prod version written (condensed format)
  • Saved with correct naming convention
  • Constellation adaptation notes added (if gold standard)
