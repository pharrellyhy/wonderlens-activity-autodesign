面向 Engineering 和 product 的 presentation

Full-pass agentic pipeline

一个 Production pass 会把 source activity idea 转成经过验证的 WonderLens package；其中包含 runtime text、image assets、downstream consumer checks、live dialogue QA、repair evidence 和 final review。

1 Autodesign 负责 source intent、package quality、runtime instructions、assets，以及所有 package-owned repairs。
2 Full production pass 在生成前必须记录 `full_pass_pipeline=true` 和 `asset_build=generate_and_curate`。
3 Image generation 是独立的 image-only phase；它只补足 activity 的视觉表达，不改写 activity 本身。
4 Fullstack-demo 和 WonderLens AI 都是 direct consumers；它们验证 package 原样运行时的行为。

为什么需要 full-pass

Package 即使通过 static structure checks，也可能没有兑现 product promise。Full pass 会在 generated activity 最容易漂移的地方增加 independent checks。

Intent 保住 source play frame Child role、sequence、required action，以及 real/reference asset promise 都要被锁住。
Runtime 让 `prod.md` 变得可执行 每个 live beat 都需要 machine-convertible runtime guidance，不能只靠顺耳的 sample line。
Assets 匹配 beat，而不是只匹配 mood board Images 必须服务实际 screen state、source promise 和 child-facing action。
Consumers 验证两条 runtime paths Fullstack-demo 和 WonderLens AI 会通过不同 conversion paths 加载同一个 package。
Repair 按 owner 路由修复 Package gaps 回到 Autodesign 修；runtime bugs 则成为 downstream follow-ups。

Source rules

更新后的 full-pass contract 以原始 workbook promise 为起点；normalized helper files 和 downstream converters 只是辅助工具，不能削弱 activity。

Workbook baseline 以 `inputs/original_activity_concepts_2026-05-29.tsv` 为准

`inputs/source_activity_concepts.md` 只用于 normalized names、mechanics、asset hints 和 capability notes。两者冲突时，以 TSV 为准，除非已经记录 product approval。

Run manifest 处理前先记录 full-pass mode

Production-scale pass 的 `run_manifest.yaml` 必须包含 `source.full_pass_pipeline: true` 和 `source.asset_build: generate_and_curate`。

Runtime contract 每个 live beat 都要有 machine-convertible instruction

Step 1、Step 2、每个 Step 3 round、Cat5 synthesis、celebration、closing 和 early exits，都需要 goal、constraint、tone、progress evidence、branch behavior 和 frame guardrail。

Cat5 shape Collection、synthesis 和 celebration 必须分开

如果存在 synthesis instructions，imports 必须单独保留它们，包括 collection criterion、accepted evidence、item-combination behavior 和 final summary format。

Text package 细节

Package writing 虽然是 text-only，但它仍然是 runtime contract。任何 image build 开始之前，writer 必须先写清 reviewer intent、可执行的 dialogue behavior、demo gates、entity binding 和 asset requirements。

spec.md Reviewer reference：source promise、adaptation rationale、scaffold fit、asset brief、asset usage timeline、extensibility notes、residual risk，以及一个且仅一个 self-evaluation scorecard。
prod.md Runtime guide：Step 1 到 Step 5、完整展开的 Step 3 rounds、screen states、child branches、follow-up policy，以及带 display/fallback behavior 的 asset IDs。
tag_block.yaml Machine contract：activity ID、category、mechanic、pillar、game style、focal attribute、progression、tags 和 entity coverage vocabulary。
recap.template.yaml + dashboard.template.yaml 面向家庭的 output templates，要和 tag block 保持一致，尤其是 focal attribute、badge/reward 和 next-step direction。
demo_support.yaml Demo gate：supported、degraded 或 unsupported status，UI template，明确的 default entity binding，runtime requirements，以及诚实的 limitation notes。
asset_manifest.yaml Text-only asset contract：独立的 runtime asset roles、style ID、screen targets、variants、nullable paths、fallback behavior 和 reference provenance requirements。

Runtime beat contract

只有当 `prod.md` 不需要 downstream 猜测，就能转成可用的 `step_instructions` 或 WonderLens AI runtime prompts，text package 才算成功。

**Runtime AI instruction:** Goal: ... Constraint: ... Tone: ... Progress evidence: ... Branch behavior: ... Frame/source guardrail: ...
Step 1 hook Step 2 mission Round 1 Round 2 Round 3 Cat5 synthesis Celebration Closing Early exit
  • Example line is not enoughExample 负责展示 tone；instruction 才承载 goal、progress evidence、branch policy 和 frame guardrails。
  • Rounds stay parseableStep 3 headings 必须使用精确的 ASCII 形态 `**Round N -- Short Scenario:**`，这样 importers 才能稳定拆分 rounds。
  • Cat5 keeps synthesis separateCollection rounds、item evidence、item-combination behavior、final story/summary 和 celebration 不能被压成一个泛泛的 step。
  • No runtime leakage除非已经证明 consumer 会剥离，否则 `RESOLVED BLOCKER` 这类 review markers 不能进入 runtime-facing `prod.md`。
  • Assets are referenced, not built`prod.md` 只引用 asset IDs 和 fallback behavior；真正的 PNGs 和 manifest paths 由后续 image phase 写入。
  • Demo gates are honestUnsupported UI-heavy mechanics 必须继续标为 unsupported；runtime-judgment Cat5 packages 要明确说明 degraded catalog 或 text-simulated behavior。

Agent roles 必须分开

Master agent 负责 orchestration，但 independent agents 要各自拥有独立的 review surface。Writer 不能成为自己 artifact 的唯一 reviewer。

Authoring side

  • Master orchestrator 控制 scope、snapshots、blockers、credentials、servers、commits 和 final report。
  • Source-intent auditor 在写作前后锁住 play frame、child role、device role、action、real assets 和 context。
  • Text-only package writer 生成 runtime prose 和 package metadata。

Asset side

  • Image-only generator 基于 accepted prompts 和 manifests 生成或 curate PNGs。
  • Image quality validator 检查 style、crop safety、beat fit 和 reference fidelity。
  • Image improver 只修 visual artifacts，不改 dialogue。

Consumer side

  • Package/import validator 检查 schema、paths、converted instruction shape 和 gates。
  • Fullstack 和 WonderLens AI validators 跑 runtime-equivalent dialogue checks。
  • Dialogue improvers 修 package-owned guidance gaps，也包括 WonderLens AI runtime prompt gaps。
  • Final reviewer 审 evidence 和 residual-risk ownership。

Asset phase

当前 activity assets 使用 flat Nordic nursery illustration style：安静的白底、克制的 boho pastel、稀疏线条、足够留白；每个 beat 或 item 单独成文件，不能内嵌 device chrome。

Style strip Generated activity assets 当前的视觉目标。
Scene beat Full-bleed square image，中心内容要 lens-safe。
Reference source 已接受的 CMA CC0 source original 和 metadata，而不是生成的近似图。
Selectable item Cat5-style screens 使用的可复用居中 object，每个文件一个。
  • 在 `asset_build=generate_and_curate` 运行前，package authoring 保持 text-only。
  • 每个 package 都要有 `activity_icon`、`intro_scene`、`rules_scene`、三个 round scenes、`celebrate_scene` 和 `closing_scene`；Cat5 或 synthesis flows 还要有 `synthesis_scene`。
  • `asset_manifest.yaml` 负责 asset IDs、roles、prompts 或 sources、variants、nullable paths 和 fallbacks；最终 PNGs 是 package-local 的 512x512 runtime assets。
  • Image QA 检查 style、crop safety、beat alignment、runtime size 可读性、reference fidelity、无重复 selectable objects，以及不能提前泄露答案。
  • Failures 只修 image-owned artifacts：prompt、source、curated redraw、crop 或 generated PNG。

Repair routing

这套 pipeline 的价值在于不掩盖 ownership。Weak package 回 Autodesign 修；如果 good package 被 runtime 忽略，就记录为 downstream follow-up。

Source intent drift Text/package repair -> source-intent re-audit

Example：constellation reveal 被改成了泛泛的数星星 activity。

Thin runtime instruction Package repair -> import and dialogue revalidation

Example：`Example AI line` 听起来不错，但 branch behavior 无法 machine-convertible。

Image mismatch Image repair -> builder -> image QA

Example：asset 看起来不错，但不匹配 screen beat 或 reference requirement。

Runtime consumer issue Downstream follow-up，除非明确纳入 scope

Example：fullstack 在 `step_instructions` conversion 中忽略了足够强的 `prod.md` instructions。

Product capability gap Blocked、degraded，或 approved minimum behavior

Example：drawing、sorting UI 或 runtime visual judgment 目前还不能诚实支持。

Credential blocker 记录具体缺失的 prerequisite

Example：credentials 不可用，所以 live provider QA 无法运行。

Acceptance gates

只有每个 gate 都有 evidence，或者 blocker 有明确 owner 且可复现，full pass 才能被 accepted。

Gate Evidence Owner

每个团队如何使用

同一份 run output 同时支持 product judgment 和 engineering contract validation。

Product view

  • Generated activity 是否保住了原始 product promise？
  • 是否有 required capability 目前还 unsupported？
  • Degraded behavior 是否可以接受，还是该 row 应该继续 blocked？
  • Image assets 是否适合 child-facing experience？
  • Dialogue transcripts 是否像预期的 activity？

Engineering view

  • Package schema 和 tag vocabularies 是否 validate？
  • `prod.md` 是否能干净地转换为 runtime instructions？
  • Fullstack-demo 和 WonderLens AI 是否保留 behavior？
  • Asset paths 是否能 resolve，且没有 unsafe fallbacks？
  • Issues 是否分配到了正确 repo 或 product decision？

如何运行

较大规模或 production-scale passes 使用这个模式：package generation、asset generation、downstream validation、dialogue QA、repair loops 和 final review 都必须在同一个 evidence-backed run 中完成。

/goal Execute goals/2026-06-01-run-full-pass-agentic-pipeline-goal.md end to end.

Scoped goal 负责 full-pass flags、delegated-agent rules、credential boundaries 和 live validation order。Manifest 仍然记录这些 flags；asset phase 在 downstream dialogue QA 之前使用 full-pass builder sequence。

python3 scripts/repair_full_pass_asset_bundle.py runs/<run_id> python3 scripts/build_activity_assets.py runs/<run_id> --mode generate_and_curate python3 scripts/validate_asset_build_outputs.py runs/<run_id> python3 scripts/validate_asset_granularity.py runs/<run_id> python3 scripts/validate_full_pass_asset_bundle.py runs/<run_id>

完成标准不是“package 存在”。真正的完成标准是：两个 direct consumers 都能加载并运行 package，而且不削弱 activity promise。
