项目进度总览

WonderLens 游戏设计系统正在分阶段构建。以下总览展示了各模块的完成情况、当前正在推进的工作，以及后续计划。每张卡片标注了对应的源文件，可直接查阅详细实现或设计文档。

已完成 设计和/或实现已完成，可直接使用。   待实现 设计文档已完成，实现是下一步优先事项。   规划中 方案已设计但尚未启动，排在更高优先级工作之后。

基础设施
定义游戏结构的核心参考文件——包括游戏风格分类体系（6 个情感支柱下的 12 种游戏风格）、设计师填写的模板、质量评估标准、以及自动化生成流程。这些是「做游戏的规则」。
docs/game_styles.md · templates.md · program.md · run.md
12 个 Entity Gold Standard
精心手工设计的标杆游戏——12 种游戏风格各一个。每个都针对特定实体设计（如狮子、香蕉、泰迪熊），是所有同类游戏的质量标尺。可以理解为每种游戏类型的「满分样板」。
designs/cat1/*_gold_*.md · designs/cat5/*_gold_*.md
18 个 Property-Bridge 模板
可复用的游戏模板，基于 AI 检测到的视觉属性（如「这个东西是圆的」→ 启动找圆形物品的游戏）。与 gold standard 不同，每个 property-bridge 模板可服务于数百种实体。12 个户外模板 + 6 个室内模板，覆盖全部游戏风格。
designs/cat5/*_property_gold_*.md · designs/cat1/*_property_gold_*.md
Style Recommender 设计文档
决定「给这个实体玩哪个游戏」的系统设计。当孩子拍照后，系统分析对话上下文，将实体匹配到最佳游戏风格，找到最兼容的现有游戏进行复用或适配。
docs/plans/2026-04-01-style-recommender.md
Entity Bridging 设计文档
处理「没有预设游戏」的实体的兜底方案。三层降级策略：(1) 通过自然对话桥接到相似实体的游戏，(2) 检测视觉属性并使用 property-bridge 模板，(3) 以丰富的对话收尾，而非强行提供低质量游戏。
docs/plans/2026-04-08-entity-bridging.md
10 维质量评估体系
每个游戏出品前必须通过的质量清单。10 个维度覆盖：技术限制、情感开场、异常处理、IB 课程对齐、年龄适配语言、对话质量、屏幕设计、实体匹配、真实趣味性（Game Feel）、以及情感辨识度（Pillar Fidelity）。
program.md (§Phase 3 rubric)
实现层 Prompt（Track C）
驱动 Style Recommender 和 Entity Bridging 实际运行的 LLM 提示词。这些是 AI 在运行时决定「玩哪个游戏」和「如何桥接实体」的具体指令。设计方案已完成，提示词文件尚未编写。
prompts/stage1_conversation_analyzer.md · prompts/stage2_style_matcher.md · constellation_map.yaml · conversation_bridge.md §5
旧版设计改写（Phase 2）
约 25 个在旧版 6 风格体系下创建的游戏，需要改写为新的 12 风格支柱体系。暂缓——12 个 gold standard + 18 个 property-bridge 模板已为上线提供了可用的游戏库。
Constellation Map 填充
建立查找表，将孩子常拍的实体（猫、狗、玩具、日用品）映射到可复用的 gold standard 游戏。这是扩展游戏覆盖面成本最低的方式——无需设计新游戏，只需建立映射关系。
收尾工作（Phase 3）
日常维护——归档旧格式文件、更新交接文档、重建完整游戏库的合并文档。

› Cat1 示例：Robot Inspector（机器人大侦探） Mystery mystery_lens

完整文件：designs/cat1/toy_robot_cat1_gold_spec.md（197 行）——以下为关键步骤的可视化展示。

实体
玩具机器人
年龄层
T1（4–6 岁）
体验支柱
Mystery — "我发现了！"
IB 概念
Form（长什么样）, Function（有什么用）
魔法时刻
每次"找到了！"的揭晓 + 最终"全部零件——破解完成！"
孩子角色
首席机器人检查官

玩法：AI 偷偷选择孩子照片上机器人的一个细节，逐步给出线索（形状→功能→位置）。孩子看着照片推理猜测。3-4 轮，难度递增。

第 1 步 — 情感开场
AI：（惊喜）"哇，看这个机器人！好多酷零件——我看到轮子、按钮、一根尖尖的天线…如果你的机器人会说话，你觉得它会说什么？"
3 种回应：理想 → "嘟嘟！" / 意外 → "它有红眼睛！" / 沉默 → 等 2 秒，轻声引导
屏幕：机器人照片居中，发光脉冲效果。齿轮形状的小星星环绕边缘。
第 2 步 — 游戏设置 + 演示
AI："我会想一个秘密零件。我给线索，你来猜！看——圆圆的…帮机器人移动的…轮子！明白了吗？现在来真的。你能在第三条线索之前猜到吗？"
屏幕：机器人照片在左，"机器人检查官"放大镜图标在右，演示动画，"案件 1"徽章。
第 1 轮 — 天线 （简单——建立信心）
AI：（神秘）"案件一！第一条线索…又细又尖——像一根小棍子往上戳！你猜是什么？"
理想："天线！" → "第一条线索就猜对了！天线接收信号——像一只听天空的小耳朵！"
猜错："手臂！" → 线索 2："从机器人头顶最上面伸出来的。像旗杆！"
沉默：→ "看机器人最顶端…"
第 2 轮 — 按钮 （引入功能概念）
AI："又小又圆…你可以用手指按下去！"
理想："按钮！" → "按钮就像小命令！" · 猜错（眼睛）：→ "可以像按门铃一样按进去的…"
第 3 轮 — 关节 （较难——隐藏部位）
AI：（悄悄话）"这个很 sneaky！藏在两个部件连接的地方…它帮机器人弯曲。"
理想："会弯的地方！" → "没有关节，机器人会硬得像根棍子！" · 说了肢体名称：→ "看手臂弯曲的地方——看到那个小连接点了吗？"
第 4 步 — 魔法时刻
AI：（激动）"首席检查官…你检查了天线——信号接收器！按钮——小指挥官！隐藏的关节——弯曲连接器！每个零件——全部破解！你正式成为首席机器人检查官！"
屏幕：金色"首席机器人检查官"徽章。破解的零件图标带勾号环绕。五彩纸屑。
第 5 步 — IB 概念收尾
AI："你观察了每个零件的样子——这就是 Form！你弄清了每个零件的用途——这就是 Function！你不只是在玩机器人。你在检查它。"
# 维度 评分 原因
D9 Game Feel PASS 每次揭晓前都有真实的不确定感。"能在第三条线索前猜到吗？"制造真实的紧张感。
D10 Pillar Fidelity PASS Mystery：线索→推理→恍然大悟！不可能被重新标记为其他支柱。
› Cat5 示例：The Wish Puff Quest（蒲公英许愿任务） Adventure quest_collector

完整文件：designs/cat5/dandelion_cat5_gold_spec.md——以下为关键步骤的可视化展示。

实体
蒲公英绒球
年龄层
T1（4–6 岁）
体验支柱
Adventure — "看我们走了多远！"
IB 概念
Change（随时间变化）, Connection（共同能力）
魔法时刻
任务完成——用所有角色共创风之冒险故事
孩子角色
许愿绒球侦察兵
任务标准
"找 3 个准备飞走的东西！"

玩法：孩子在户外寻找符合任务标准的物品。每个发现都会被评估（"它准备好飞了吗？"），由孩子命名，最终编织成一个合作创作的故事。

第 1 步 — 情感开场
AI：（屏息惊叹）"哦——蒲公英绒球！看那些小种子——每颗都有自己的小降落伞！风一吹，它们就像小小的愿望一样飞走了。你看过它们飘吗？"
3 种回应：理想 → "看过！飞好远！" / 意外 → "是花！" / 沉默 → "看到那些白色小东西了吗？它们在等风…"
屏幕：蒲公英照片带发光光晕，动画种子飘向上方，"任务：0/3"
第 2 步 — 任务说明
AI：（探险家语气）"你现在是许愿绒球侦察兵！任务：找 3 个准备飞走的东西。能被风吹走的叶子！能飘起来的种子！每找到一个，告诉我它让你想到什么——我们给它起个任务名字！"
屏幕：任务卡，"许愿绒球侦察兵"徽章，3 个空任务槽，风之指南针。
第 1 轮 — 发现 + 评估 + 命名
（孩子拍了一片薄叶子）
AI：（仔细看）"一片叶子！看它多薄多轻。风一吹…它能飞走吗？（戏剧性停顿）…能！又薄又轻——完全准备好飞了！任务物品 #1！它让你想到什么？"
理想："小船！" → "小船就是它的名字！任务：1/3！" · 意外："它是绿色的！" → "绿色又薄——飞起来正好！叫它什么？" · 沉默：→ "它让我想到一片小翅膀…你呢？"
屏幕：叶子照片放入任务槽 1。"准备飞了吗？是！"印章。任务计数：1/3。
第 2 轮 — 第二个发现
（孩子拍了一根羽毛）
AI："一根羽毛——轻如空气，微风就能飘走——（戏剧性停顿）没错！任务物品 #2！它让你想到什么？"
孩子命名 → "任务：2/3！还差一个！"
第 3 轮 — 最后发现 + 任务完成！
（孩子拍了一颗枫树翅果）
AI："枫树翅果！落下来的时候会旋转！准备好飞了吗？…当然！任务完成——3/3！"
屏幕："任务完成！"横幅，3 张照片带角色名字，庆祝动画。
第 4 步 — 魔法时刻：风之冒险故事
AI：（讲故事语气）"现在…绒云、小船和旋转枫果都在一起了！一阵大风来了，呼——他们全部飞起来了！绒云带路，小船乘着上升气流…"→ AI 用孩子的角色名和细节讲述共创冒险故事。
屏幕：风之冒险路径图，所有角色在旅途中。"任务完成！"
第 5 步 — IB 概念收尾
AI："你的蒲公英从黄色花朵变成了飞行绒球——这就是 Change！你找到的所有东西都能飞，就像蒲公英一样——这就是 Connection！你获得了风之任务徽章！"
屏幕：金色"风之任务徽章"，蒲公英剪影，角色照片，"Change"和"Connection"概念词。
# 维度 评分 原因
D9 Game Feel PASS 任务标准为每次发现创造真实的紧张感。"准备好飞了吗？"的判定是真正的游戏时刻。
D10 Pillar Fidelity PASS Adventure：任务标准 + 任务进度 = 明确的 Adventure，不可能是其他支柱。

3B. Property-Bridge 模板 Tier P

定义：由检测到的视觉属性（如"圆形"、"红色"、"金属"）参数化的游戏，非针对特定实体。适用于具有该属性的任何实体。

质量：良好（7-8/10）。有结构化的游戏机制，但非针对特定实体定制。

当前库存：18 个 property-bridge 模板（12 个 Cat5 + 6 个 Cat1）

Cat5 — 户外 / 收集探索（12 个模板）

模板 Style 体验支柱 属性 属性如何驱动游戏 示例

Movers Quest quest_collector Adventure wheels, legs, wings 属性成为任务标准——"找能动的东西！" 玩具车（有轮子）→ "找 3 个能动的东西！"
Material Lab field_experiment Discovery metal, wood, plastic 属性成为假设——"大部分东西是 [材质] 还是不是？"AI 通过照片评估。 叉子（闪亮金属）→ "这里大部分是金属还是不是？"
Size Experiment field_experiment Discovery bigger/smaller（参照物） 实体成为大小参照——"大部分东西比 [这个] 大还是小？" 松果 → "比你的手大还是小？"

Shiny Experiment field_experiment Discovery shiny, dull/matte 属性成为假设——"大部分东西是亮的还是暗的？"AI 通过照片评估。 门把手（反光）→ "这个房间里亮的多还是暗的多？"
Old vs New field_experiment Discovery worn/faded, bright/clean 属性成为假设——"大部分东西看起来旧还是新？" 锈迹斑斑的铁门（有磨损）→ "周围看起来旧的多还是新的多？"
Pattern Trail mystery_trail Mystery stripes, spots, zigzag, lines AI 给出关于图案事物的谜语线索。找到 3 个后，揭示隐藏的联系。 条纹衬衫 → AI 谜语引导找到斑点、锯齿、线条——"图案无处不在！"
Texture Mix mix_lab Creation rough, smooth, fuzzy, bumpy AI 根据视觉纹理赋予"超能力"。孩子将 2 个超能力组合发明新东西。 树皮（看起来粗糙）→ 粗糙=抓握力, 光滑=速度 → "发明一个又能抓又快的东西！"

Living Rescue rescue_team Nurture alive, plant, growing AI 从生物照片中识别"需求"。孩子提出关爱方案。被救助的事物互相帮助。 萎靡的植物 → "找 3 个需要帮助的生物！" → 互助圆圈

模板 Style 体验支柱 属性 属性如何驱动游戏 示例
Detail Detective mystery_lens Mystery color, shape, material 属性成为线索类别——AI 选择实体上该属性类型的细节，逐步给出线索。 彩色卡车（多种颜色）→ "你的卡车上有个不是红色的东西…" → 孩子逐个猜出每种颜色
What-If Workshop inventor_workshop Creation 任何检测到的属性 属性的反义词成为改造提示——"你的 [entity] 是 [属性]，如果变成 [反义] 呢？" 闪亮金属勺子 → "如果变成毛茸茸的呢？""如果变成软软的呢？" → 超级勺子！
Property Predictor prediction_lab Discovery shape, size, material 属性生成预测场景——"当带有 [属性] 的 [entity] 遇到 [情境] 会怎样？" 圆球 → "滚到桌子边缘——会掉下去还是停住？" → 孩子预测 → 揭晓 + 得分
Time Shifter time_traveler Adventure color, size, condition 当前属性成为时间旅行的一站——"现在是 [属性]，之前是什么样？" 黄色香蕉 → "几个月前它还是一朵绿色的花！" → 颜色变化的时间轴
Fix-It Station care_station Nurture worn, faded, dusty, torn 磨损属性触发第一个关爱需求。需求逐步升级：身体 → 情感 → 复合问题。 褪色的布偶 → "它的颜色在变淡！怎么帮它？" → 可见的修复变化
Property Performer voice_stage Performance shape, size, color 属性成为表演主题——孩子扮演实体，用表达方式展现这个属性。 圆球 → "你的球好圆！给我听你最大的滚动声！" → 观众疯狂鼓掌

3C. Constellation 适配游戏 Tier B

定义：将一个实体 gold standard 游戏复用于语义相似的实体。无需创建新的游戏文件——适配在运行时完成。

示例：小猫 → 没有 gold standard → 小猫属于狮子的 constellation → 适配狮子的游戏：将"吼叫"替换为"喵"，"丛林"替换为"社区"。

何时扩展：在 constellation_map.yaml 中添加条目。无需设计游戏——只需做映射。

4 如何设计 Entity Gold Standard 已完成 ›

分步指南

  1. 选择实体 + 类别 + 层级 + 体验支柱 + style — 实体来自 IB 思维导图，类别基于使用场景，层级基于年龄，体验支柱基于期望的情感体验
  2. 阅读实体映射 YAML — 记录 IB 主题、主要关键概念（必须使用 ≥1 个）、tier_guidance、选择 2-3 个锚定维度
  3. 阅读基础模板 + 体验支柱覆盖层，来源：templates.md
  4. 研究对应 style 的 gold standard — 阅读 spec 和 prod 两个版本
  5. 填写创意变量 — 通用变量：{metaphor}、{role_title}、{escalation_axis}。体验支柱专属变量见 templates.md。必须是全新的。

  7. 对照 10 个维度进行自评 — 修复失败项，重新评估直到全部 PASS
  8. 编写 prod 版本 — 项目列表→表格，精简第 2 轮及以后的内容，移除评分卡
  9. 使用正确命名保存 — {entity}_cat{N}_gold_spec.md + _gold_prod.md

5 如何设计 Property-Bridge 模板 已完成 ›
相关文件: docs/plans/2026-04-08-entity-bridging.md (§Layer 2) · templates.md

流程与 entity gold standard 相同，但有以下关键差异：

方面 Entity Gold Standard Property-Bridge
触发条件 特定实体（如"lion"） 任何具有被检测到的 {property} 的实体
映射来源 实体映射 YAML 或"无" property-bridge
内容驱动 实体的独特特征 被检测到的属性 + 游戏含义
复用性 仅 1 个实体 多个实体

允许的属性（AI 可在照片中识别）

属性 示例 可靠性
颜色 red, blue, green, yellow, brown 非常高
形状 round, pointy, flat, long, curvy 高
材质类型 metal, wood, plastic, fabric, stone 高
尺寸（相对） 比手大/比手小 高
天然 vs 人造 树 vs 长椅 高
有生命 vs 无生命 植物、动物 vs 物体 高
图案 stripes, spots, zigzag 高
光泽度 反光 vs 哑光 高
老化/磨损 锈蚀、裂纹 vs 明亮、干净 中高
可见纹理 粗糙的树皮、光滑的玻璃 中
运动能力 轮子、腿、翅膀 中
声音潜力 棍子=敲击、树叶=沙沙声 创造性推断

禁止的属性

属性 禁止原因
实际触感（触觉） 无法通过照片触摸
重量 无法通过照片称量
温度 无法通过照片感知
实际声音 无法通过照片听到
气味/味道 无法通过照片感知
柔韧性 在静态图像中不明确
field_experiment 特殊规则：AI 作为评估者
AI 从照片中评估属性并宣布结果。孩子不需要自行报告。这保留了"我猜对了吗？！"的惊喜感。
示例：AI 说"让我仔细看看……我看到它是亮闪闪的！"（而不是"它是亮的还是暗的？"）
6 Quality Rubric（质量评估维度）：10 个维度 已完成 ›
相关文件: program.md (§Phase 3 rubric)

每个游戏必须通过全部 10 个维度。

# 维度 PASS 条件… FAIL 条件…
D1 V1 技术合规 不依赖 OCR、面部检测、IMU 或前后对比 设计依赖被屏蔽的能力
D2 Hook 与过渡 步骤 1 以情感共鸣开场，自然展开 步骤 1 测试知识或突然分配任务
D3 边缘情况覆盖 每一步有 3 个分支（理想、意外、沉默） 缺少分支或忽略意外回应
D4 IB 完整性 指定 1-2 个关键概念。定义 KUD。标注 ATL 技能。 概念缺失或与孩子的行为不匹配
D5 层级适切性 语言匹配层级。T0: ≤5 词。T1: 5-8 词。T2: 复杂。 语言对该层级来说太高级或太简单
D6 对话具体性 所有 AI 台词都是带语气标记的实际对话 抽象描述如"AI 引导…"
D7 屏幕与 UI 每一步都有具体的屏幕描述 模糊或缺少屏幕描述
D8 实体映射对齐 （仅限映射驱动型）概念来自映射 概念无法追溯到映射
D9 游戏感 真正的不确定性 + 令人满意的揭晓 + 赌注 感觉像结构化的问答，没有惊喜
D10 体验支柱忠实度 盲测读者能识别出体验支柱。情感弧线匹配。 换个标签也不会觉得违和

7 游戏选择流程 设计完成 ›
相关文件: docs/plans/2026-04-01-style-recommender.md · docs/plans/2026-04-08-entity-bridging.md
孩子拍摄实体 ↓ 实体在 IB 思维导图中？──是──→ 实体有 gold standard 游戏？ │ ├── 是 → Tier A（gold standard） │ └── 否 → Style Recommender → Tier B（适配） │ → 或 Tier P │ └── 否 → Entity Bridging 系统 ├── Constellation 匹配（≥ 0.7）？ │ └── 是 → 在对话中桥接 → Tier A/B │ ├── Property bridge 可用？ │ └── 是 → Tier P（property-bridge 模板） │ └── 无匹配 → 纯对话（无游戏）

Entity Constellation 桥接规则

  1. 每次对话只能尝试一次桥接——如果孩子忽略了，绝不重试
  2. 尊重原始实体——桥接是增加关联，绝不忽视原实体
  3. 类别必须匹配——不要将室内实体桥接到户外游戏
  4. 最低 constellation 分数 0.7
  5. 游戏引用两个实体——"既然你的小猫是狮子的亲戚……"

桥接类型（从强到弱）

  1. 同一分类体系（小猫 → 狮子）
  2. 部分-整体（秋千 → 游乐场）
  3. 同一领域（郁金香 → 向日葵）
  4. 共享功能（玩具车 → 自行车）
  5. 相同角色（毛绒兔子 → 泰迪熊）
  6. 伴随关系（牙刷 → 牙刷架）
  7. 共享栖息地（池塘青蛙 → 金鱼）

任务队列格式

- [ ] goldfish + category 1, tier=T1, mapping=animals_goldfish,
            start=warm+cold, scene=child photographs their pet goldfish
字段 必填 说明
entity 是 实体名称
category 是 1（Cat1）或 5（Cat5）
tier 是 T0、T1 或 T2
pillar 否 体验支柱（省略时自动推断）
style 否 Game Style（省略时自动推断）
mapping 否 实体映射 ID — 启用映射驱动设计
start 否 warm+cold 表示双桥接，或省略表示仅冷启动
scene 否 触发场景（省略时自动生成）

设计循环

步骤 1：解析任务 → 提取 entity、category、tier、pillar、style、mapping ↓ 步骤 1.5：加载实体映射 YAML（如果指定了 mapping=） → 提取 IB 主题、关键概念、tier_guidance → 选择 2-3 个锚定维度 ↓ 步骤 2：加载基础模板 + 体验支柱覆盖层 → 基于映射数据构思创意变量 ↓ 步骤 3：生成完整活动设计（spec 格式） → 5-6 步，完整对话，3 个响应分支，屏幕描述 ↓ 步骤 4：对照 10 个维度自我评估 → 修复失败项，重新评估直到全部 PASS ↓ 步骤 5：保存 spec → designs/{entity}_cat{N}_gold_spec.md ↓ 步骤 5.5：将 spec 转换为 prod（应用 transform.md 规则） → 保存 prod → designs/{entity}_cat{N}_gold_prod.md ↓ 步骤 6：记录结果 → results.tsv ↓ 步骤 7：标记任务 [x] 完成 ↓ 步骤 8：Git 提交（spec + prod + results.tsv + assignments.md） ↓ 步骤 9：下一个任务 → 重复

映射驱动型 vs 非映射型

方面 映射驱动型 非映射型
实体数据 完整 IB YAML（主题、概念、tier_guidance） 无——智能体使用通用知识
桥接类型 热启动 + 冷启动（步骤 1a + 1b） 仅冷启动（步骤 1b）
D8 评估 评分 N/A
质量 更高——基于研究过的 IB 数据 良好——但 IB 基础较弱

批量处理技巧

  • 并行化：不同体验支柱可在不同智能体上同时运行
  • 先做范例：在规模化之前，每种 style 先生成 1 个 gold standard
  • 混合层级：交替使用 T0/T1/T2 以增加多样性
  • 多样化实体：每种 style 使用不同类型（动物、物体、场所）的实体
