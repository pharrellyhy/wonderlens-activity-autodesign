Engineering and product presentation

Full-pass agentic pipeline

A production pass turns a source activity idea into a validated WonderLens package with runtime text, image assets, downstream consumer checks, live dialogue QA, repair evidence, and final review.

1 Autodesign owns source intent, package quality, runtime instructions, assets, and package-owned repairs.
2 Full production passes record `full_pass_pipeline=true` and `asset_build=generate_and_curate` before generation.
3 Image generation is a separate image-only phase. It supports the activity, it does not rewrite it.
4 Fullstack-demo and WonderLens AI are direct consumers. They validate package behavior as provided.

Why the full pass exists

A package can pass static structure checks and still fail the product promise. The full pass adds independent checks at every place where a generated activity can drift.

Intent Preserve the source play frame Child role, sequence, required action, and real/reference asset promises stay locked.
Runtime Make `prod.md` executable Every live beat needs machine-convertible runtime guidance, not only pleasant sample lines.
Assets Match the beat, not the mood board Images must support the actual screen state, source promise, and child-facing action.
Consumers Validate both runtime paths Fullstack-demo and WonderLens AI load the same package through different conversion paths.
Repair Route fixes by owner Package gaps return to autodesign. Runtime bugs become downstream follow-ups.

Source rules

The updated full-pass contract starts from the original workbook promise, then treats normalized helper files and downstream converters as support tools. They cannot weaken the activity.

Workbook baseline `inputs/original_activity_concepts_2026-05-29.tsv` controls

Use `inputs/source_activity_concepts.md` only for normalized names, mechanics, asset hints, and capability notes. If the two disagree, the TSV wins unless product approval is recorded.

Run manifest Record full-pass mode before processing

`run_manifest.yaml` must include `source.full_pass_pipeline: true` and `source.asset_build: generate_and_curate` for production-scale passes.

Runtime contract Every live beat has a machine-convertible instruction

Step 1, Step 2, every Step 3 round, Cat5 synthesis, celebration, closing, and early exits need goal, constraint, tone, progress evidence, branch behavior, and frame guardrail.

Cat5 shape Collection, synthesis, and celebration stay distinct

Imports must preserve synthesis instructions separately when present, including collection criterion, accepted evidence, item-combination behavior, and final summary format.

Agent roles stay separate

The master agent orchestrates, but independent agents own disjoint review surfaces. A writer should not be the only reviewer of its own artifact.

Authoring side

  • Master orchestrator controls scope, snapshots, blockers, credentials, servers, commits, and final report.
  • Source-intent auditor locks play frame, child role, device role, action, real assets, and context before and after writing.
  • Text-only package writer creates runtime prose and package metadata.

Asset side

  • Image-only generator produces or curates PNGs from accepted prompts and manifests.
  • Image quality validator checks style, crop safety, beat fit, and reference fidelity.
  • Image improver repairs visual artifacts without changing dialogue.

Consumer side

  • Package/import validator checks schema, paths, converted instruction shape, and gates.
  • Fullstack and WonderLens AI validators run runtime-equivalent dialogue checks.
  • Dialogue improvers repair package-owned guidance gaps, including WonderLens AI runtime prompt gaps.
  • Final reviewer audits evidence and residual-risk ownership.

Asset phase

Current activity assets use a flat Nordic nursery illustration style: quiet white backgrounds, restrained boho pastels, sparse linework, generous negative space, one file per beat or item, and no baked device chrome.

Cat5 cards/items guide selection and capture. They do not replace real camera capture or the downstream `photo_id` handoff.

Style strip Current visual target for generated activity assets.
Scene beat Full-bleed square image with lens-safe central content.
Reference case Real-world facts need accepted provenance, not random approximation.
Selectable item One reusable centered object per file for Cat5-style screens.
  • Package authoring stays text-only until `asset_build=generate_and_curate` runs.
  • Every package gets `activity_icon`, `intro_scene`, `rules_scene`, three round scenes, `celebrate_scene`, `closing_scene`, and `recap_scene`; Cat5 or synthesis flows also get `synthesis_scene`.
  • `asset_manifest.yaml` owns asset IDs, roles, prompts or sources, variants, nullable paths, and fallbacks; final PNGs are package-local 512x512 runtime assets.
  • Image QA checks style, crop safety, beat alignment, readable runtime size, reference fidelity, no duplicate selectable objects, and no premature answer reveal.
  • Failures repair image-owned artifacts only: prompt, source, curated redraw, crop, or generated PNG.

Repair routing

The pipeline is useful because it refuses to hide ownership. A weak package is fixed in autodesign. A good package ignored by a runtime becomes a downstream follow-up.

Source intent drift Text/package repair -> source-intent re-audit

Example: a constellation reveal becomes a generic star counting activity.

Thin runtime instruction Package repair -> import and dialogue revalidation

Example: `Example AI line` is pleasant, but no branch behavior is machine-convertible.

Image mismatch Image repair -> builder -> image QA

Example: asset looks nice but does not match the screen beat or reference requirement.

Runtime consumer issue Downstream follow-up, unless explicitly in scope

Example: fullstack ignores strong `prod.md` instructions during `step_instructions` conversion.

Product capability gap Blocked, degraded, or approved minimum behavior

Example: drawing, sorting UI, or runtime visual judgment is not honestly supported yet.

Credential blocker Record exact missing prerequisite

Example: live provider QA cannot run because credentials are unavailable.

Acceptance gates

What each team gets

The same run output supports product judgment and engineering contract validation.

Product view

  • Does the generated activity preserve the original product promise?
  • Is a required capability unsupported today?
  • Is degraded behavior acceptable, or should the row stay blocked?
  • Are image assets appropriate for the child-facing experience?
  • Do dialogue transcripts feel like the intended activity?

Engineering view

  • Do package schema and tag vocabularies validate?
  • Does `prod.md` convert into runtime instructions cleanly?
  • Do fullstack-demo and WonderLens AI preserve behavior?
  • Do asset paths resolve with no unsafe fallbacks?
  • Are issues assigned to the right repo or product decision?

How to run it

Use this mode for larger or production-scale passes, where package generation, asset generation, downstream validation, dialogue QA, repair loops, and final review must happen in one evidence-backed run.

/goal Execute GOAL.md end to end using run.md with asset_build=generate_and_curate full_pass_pipeline=true.

The manifest must record both flags, and the asset phase uses the full-pass builder sequence before downstream dialogue QA.

python3 scripts/repair_full_pass_asset_bundle.py runs/<run_id> python3 scripts/build_activity_assets.py runs/<run_id> --mode generate_and_curate python3 scripts/validate_asset_build_outputs.py runs/<run_id> python3 scripts/validate_asset_granularity.py runs/<run_id> python3 scripts/validate_full_pass_asset_bundle.py runs/<run_id>

The completion standard is not "the package exists." The completion standard is that both direct consumers can load and exercise the package without weakening the activity promise.
