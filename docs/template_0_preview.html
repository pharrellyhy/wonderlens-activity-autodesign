§ 01 / At a glance

Three internal asks,
three structural answers.

Each concern you raised maps to a dedicated slot in Template 0. The goal is to make the things you care about first-class at the template layer — so no design can ship without naming them.

01
"The tag system must be explicit — so it can bridge the upstream app, a child recap screen, and a parent growth path."

A first-class tag block

A required header on every activity exposing entity, category, attributes, Key Concepts, Related Concepts, and ATL skills — the exact hierarchy you asked for. The same block also feeds the game selection pipeline upstream and the child + parent surfaces downstream.

→ see § 04 Tag block · § 05 Bridges
02
"How do we design progression — same topic, a bit harder each time?"

A shared progression axis

Activities declare a topic_axis and a difficulty level 1–3. The seven axes are derived from the seven IB Key Concepts already in the playbook, so progression rides the same thinking lens the rest of the framework already uses.

→ see § 07 Progression
03
"Same game adapted to age by interaction, or different activities per age?"

Adapt within, split across

A design targets one primary tier and optionally flexes to a neighbor via five explicit dials (AI role, language, task depth, concept focus, caregiver role). Crossing two tiers means siblings on a shared axis — not one stretched design.

→ see § 06 Tier adaptation
§ 02 / The spine

Five beats, invariant.

Every category — Cat1, Cat5, and the four still to come — rides the same 5-beat emotional arc. What changes between categories is the medium of each beat (verbal dialogue vs. photo collection vs. physical exploration), not the beat itself.

The 6 experience pillars — each pillar defines the emotional shape of beats 2–4
Mystery "I figured it out!"
Creation "I made this!"
Performance "They loved it!"
Discovery "Was I right?!"
Adventure "Look how far we went!"
Nurture "I helped!"
Beat 01

Transition Bridge

Photo → emotional or imaginative reaction. Never knowledge-testing.

Beat 02

Frame + Role

Assign role title + metaphor. State the mission in pillar-specific language.

Beat 03

Core Loop

N rounds of the pillar-specific mechanic. {escalation_axis} must ride the declared progression axis.

Beat 04

Magic Moment

Pillar-specific synthesis. Emotional climax. Blind reader should identify pillar from this beat alone.

Beat 05

Celebration + Concepts

Name earned Key Concepts + Related Concepts as praise — never as knowledge check.

Cat1 and Cat5 become thin overlays that only override 8 fields: beat medium, round count, camera use, setting, step count, core mechanic, and the anchor-dimension priorities. Everything else — pillar overlays, creative variables, entity adaptation checklist — is inherited from Template 0.

§ 04 / The tag block

Every activity starts with a handshake.

This YAML block is required at the top of every design. It's the contract between design and the upstream app, the child's recap screen, and the parent's growth path dashboard. The highlighted fields are new or newly-promoted to first-class status.

Why these slots — split by the surface they feed

Downstream consumers split cleanly into two surfaces. Child recap needs intuitive, visible, "what I did" labels. Parent growth path needs the IB framework + cross-subject classification that positions the activity inside the curriculum. Some tags feed both; most feed one or the other.

Child-facing · intuitive achievement
  • role_titleThe badge the child earned. Verb-forward, specific to what they did.
  • pillarThe emotional mode (Mystery, Creation, …) — becomes the mission frame.
  • related_concepts2–4 child-friendly concept words (Pattern, Camouflage, …) shown as earned chips.
  • highlight_momentNEW. Runtime-generated one-liner: "you spotted / you figured out / you cared for…". A concrete trophy, not a summary.
  • attributesObservable properties the game invoked — echoes what the child actually noticed.
Parent-facing · IB + subject framework
  • key_concepts1–2 of the 7 IB Key Concepts. The "thinking lens" parents use to track depth. Distinct tree from related concepts.
  • atl_skills2–3 IB ATL skills practiced. Parents can read them as cross-activity meta-skill growth.
  • transdisciplinary_themeWhich of the 6 IB TD themes the activity lives under.
  • subject_tagsNEW. Cross-subject classification (science / language / art / math / social-emotional). Lets parents filter or compare with school subjects — distinct from the child view.
  • progressionThe axis + level the activity targeted, plus the next step. Drives the growth-path ladder visualization.
  • tier · tier_variantsExplicit audit trail — which age register the activity was tuned to.
caregiver_role is a list, not a single value — T0 is [scaffold], T1 adds co-explorer, T2 includes all three with decreasing weight. Soft guidance for now; the runtime doesn't refuse to start if caregiver isn't present.
Matchability — three tags that keep a small catalog portable
entity_class
Ordered list from specific to general. The matcher rolls up when exact-entity matches miss. See docs/entity_ontology.md for the canonical class hierarchy.
entity_binding
bound = activity needs this specific entity. parameterized = entity is a slot; works for any member of the class. agnostic = cognitive drill, works on anything.
tier_range
primary = preferred tier. span = accepted tiers. elasticity: strict = primary only; neighbor = ±1 tier; broad = all three.
Tag lifecycle — catalog-time vs instance-time
Catalog-time · author declares
entity_class, entity_binding, tier_range, pillar, key_concepts, atl_skills, subject_tags, transdisciplinary_theme, category, progression.topic_axis
Instance-time · runtime populates
entity (what was seen), tier (child's actual tier), highlight_moment (runtime one-liner), caregiver_role (observed), progression.difficulty_level (rung reached), progression.next_step_hint
Rule of thumb: if the value would differ between two children playing the "same" activity, it's instance-time.

From photo to activity

When a child photographs an object, the game selection pipeline (playbook §7) decides which activity plays. It reads the tag block to rank three tiers of match — Tier A > Tier B > Tier P > conversation-only. Every tag block field earns its place by feeding one of them.

Tier A · 10/10

Entity gold standard

Entity has its own hand-designed gold standard. The pipeline does an exact match on entity + category. 12 exist today (playbook §3A).

Reads from tag block
  • entity + category → exact lookup
  • tier → language + task register
  • pillar + style → game mechanic
  • key_concepts → confirms KUD alignment
Tier B · 8/10

Constellation-adapted

No direct gold standard, but a semantically similar entity does have one (kitten → lion, swing → playground). Runtime adaptation, no new design needed — just constellation_map.yaml.

Reads from tag block
  • entity + entity_type → taxonomy / shared-role match
  • category → must match (no indoor → outdoor bridging)
  • attributes → shared affordances
  • min constellation score 0.7
Tier P · 7–8/10

Property-bridge template

No gold standard and no constellation match — but a detected visual property triggers a parameterized template. 18 exist today (playbook §3B). Works for hundreds of entities at once.

Reads from tag block
  • attributes → the key field for matching
  • category → which template set (6 Cat1, 12 Cat5)
  • tier → template's language flex
  • property must be visually verifiable from photo
Why this matters for Template 0. By making attributes, entity_type, and tier first-class in the header, every gold standard designed under Template 0 becomes automatically discoverable by the selection pipeline. The playbook describes the pipeline; the tag block is what lets every new design plug into it on day one instead of being retrofitted later.

§ 07 / Progression · ask #2

Seven axes,
from the seven Key Concepts.

Seven progression axes, one per IB Key Concept — derived directly from the seven the framework already uses (playbook §2, program.md §1.4). Each axis is a trajectory through one concept. The thinking lens and the progression ladder share the same seven names, so nothing new has to be memorized.

Derivation. Form / Function / Causation / Change / Connection / Perspective / Responsibility are already how every activity picks its "thinking lens" today. Promoting them to progression axes means a T1 Form-L2 design and a T0 Form-L1 design sit on the same ladder — the child's growth story reads as moving along a concept, not graduating from one game to another. Color, shape, texture, size are all sub-cases of Form; they specialize a rung, they are not axes themselves.

What L1 / L2 / L3 mean on every axis
L1 Notice Observe, name, or identify one instance of the concept.
L2 Extend Add more examples, more parts, more steps, or a related viewpoint.
L3 Reason Explain why, predict what-if, or state the rule that ties the whole set together.
The exact wording differs per axis — Form's L1 is "name a dominant attribute", Connection's L1 is "match one shared property", Responsibility's L1 is "notice a need" — but the cognitive move underneath is identical: L1 first observation → L2 elaboration → L3 explanation. L1/L2/L3 are a cognitive depth axis and are orthogonal to T0/T1/T2, which adapts language register to age. A 4-year-old and a 7-year-old can both be on Form-L2; the rung is the same, the words differ.

Form — attribute description

From naming one observable attribute to explaining why the attribute matters. Color, shape, texture, size, pattern all live here as specializations of a single rung.

Level 01
Name a dominant attribute

Child picks one observable property and names it — color, shape, size, texture, pattern, or count.

"The ladybug is red." / "It is round." / "It has spots."
Level 02
Name multiple, locate each

Child names 2–3 attributes and places each on a specific part of the entity.

"Red body, black spots, tiny black legs."
Level 03
Explain why the attribute exists

Child reasons about purpose — warning, camouflage, ripeness, signaling, protection.

"Red warns birds not to eat it."

Function — part & function

From pointing at parts to predicting what happens when a part is missing. The classic part–whole trajectory.

Level 01
Point to and name the parts

Child names the visible parts of the entity without explaining what they do.

"Wheels. Door. Window. Roof."
Level 02
Explain what each part does

Child connects each part to its role.

"Wheels roll. Door opens so people get in. Window lets you see out."
Level 03
Predict what happens without a part

Child reasons counterfactually about a removed or broken part.

"No wheels? It can't move. No door? Nobody gets in."

Causation — cause & effect

From describing what happened to explaining the underlying mechanism.

Level 01
Describe what happened

Child narrates the observed event without inferring cause.

"The flower closed up."
Level 02
Predict what will happen next

Child projects a future outcome from the current state.

"If I bring it into the sun, it will open again."
Level 03
Explain the mechanism

Child gives a causal story for the observed behavior.

"The petals need sunlight to stay open — they close to save energy at night."

Change — sequence & transformation

From ordering two moments to telling a full transformation journey with connecting causes.

Level 01
Order two moments

Child places two states in correct temporal order.

"First the green banana. Then the yellow banana."
Level 02
Fill in before & after

Child adds a missing third moment to complete a sequence.

"Before: green. Now: yellow. After: brown and soft."
Level 03
Tell a transformation journey

Child narrates the whole arc with connecting causes between stages.

"The seed drank rain, grew a stem, made leaves, opened a flower, then dropped new seeds."

Connection — pattern & grouping

From matching one shared property to articulating the rule that connects a whole set. Pattern recognition and classification both live here.

Level 01
Match one shared property

Child finds a second item that shares a single visible property with the first.

"This leaf is also green, like the apple!"
Level 02
Find 3+ examples of a property

Child extends the match to a small collection and groups them.

"Three spotted things: the ladybug, the ball, the cow."
Level 03
Identify the rule that connects the set

Child articulates the property or purpose the whole group shares — and why the grouping is meaningful.

"They all have spots. Spots help them hide from predators."

Perspective — point of view

From attributing a simple feeling to generating dialogue between two entities — empathy as a ladder.

Level 01
Attribute a feeling

Child assigns a simple emotion to the entity.

"The teddy is sleepy."
Level 02
Imagine the entity's view of you

Child reverses viewpoint and imagines what the entity sees / feels about the child.

"Teddy thinks I'm big and warm."
Level 03
Generate dialogue to a third party

Child voices the entity speaking about something other than itself.

"Teddy tells Bunny: 'My child is gentle and tucks me in.'"

Responsibility — care & impact

From noticing a need to reasoning about long-term impact of care or neglect. Natural home for the Nurture pillar.

Level 01
Notice a need

Child observes that the entity has a need — dirty, dry, lonely, broken.

"The flower is droopy."
Level 02
Propose a care action

Child names a concrete action that would help.

"I'll give it water and move it near the window."
Level 03
Reason about long-term impact

Child explains what happens over time with care, and what happens without it.

"If I water it every day, it grows. If I forget, the leaves dry and fall off."

Promotion & demotion — when to move the child between rungs

Two sub-questions from the review: (1) is L1/L2/L3 granular enough? and (2) when should the runtime move a child up or down? Answer: the three rungs are the commitment; within each rung the task depth stretches via the 5 tier dials. Movement between rungs starts with concrete quantified triggers — we get something measurable into production first, then replace the thresholds with a per-child pace model once there's enough session data to train one.

Within a single activity
round-by-round · in-session escalation
↑ promote Child answered the current rung correctly in 2+ rounds without prompt-repetition, or spontaneously produced an L+1 response (e.g. gave a reason when asked to name).
· hold Mixed or slow but on-topic. Stay at current rung, vary the exemplar instead.
↓ demote Child did not complete the current rung after 2 attempts (silence > 6s, off-topic, or repeats the prompt back). Drop one rung, same axis — never abandon the axis mid-activity.
Across activities
session-by-session · axis progression
↑ promote Child finished at the current rung in 3 consecutive activities on the same axis — next activity on that axis defaults to rung + 1.
· hold Fewer than 3 consecutive successes, or mixed success across axes. Keep rung, rotate entities and pillars for variety.
↓ demote Child needed an in-activity demote in 2 consecutive sessions on the same axis — next activity on that axis defaults to rung − 1 (floor L1).
Dignity rule A demote is never surfaced as failure. Language switches from "let's try again" to a new framing: "let me show you what I notice first". The child experiences variety, not regression.
Sibling jump If the child has hit L3 on one axis, the next activity can jump to L1 on a related axis (Form L3 → Connection L1) instead of forcing L3 elsewhere. Keeps novelty without pushing depth faster than the child can absorb.
Personalization hook These thresholds are a progression_policy object in the tag block's upstream config — not hardcoded. V1 uses the defaults above; once ≥ ~200 sessions per child are logged, swap in a per-child pace model without any activity design change.

§ 09 / Decisions log

Four calls, four answers.

These were open questions before the internal review. The answers are now locked in and drive the shape of Template 0.

01
How many canonical topic_axis values do we seed, and where do they come from?
Seven, one per IB Key Concept — Form, Function, Causation, Change, Connection, Perspective, Responsibility. The earlier draft picked overly-specific axes (e.g. "color_naming") that were really just sub-cases of Form. Deriving directly from the Key Concepts means the thinking lens and the progression ladder share the same seven names.
Decided
02
Does tier_variants live in the header, or is it emergent at runtime?
In the header. Explicit > implicit for audit + parent-dashboard purposes. Authors pay a small authoring cost in exchange for traceability.
Decided
03
Are key_concepts and related_concepts one tree or two?
Distinct trees. Key Concepts are the IB "thinking lens" (7 universal). Related Concepts are discipline-tagged child-facing badges. Parent dashboard can display them in two separate columns.
Decided
04
Is caregiver role at T0 a hard gate or soft guidance?
Soft guidance for now. T0 designs default to caregiver_role: scaffold, but the system doesn't refuse to start if caregiver isn't present. Can be tightened into a gate after we watch real usage.
Decided
"One skeleton the whole product can grow into — so the tag system, the progression ladder, and the parent-facing growth path all read from the same spine."
