§ 01 / The matching problem

A child photographs
something. Now what?

The child's photo is the only signal. The system has to pick one activity design out of many, run it, and make it feel like that game was written for this thing. Four properties of the real product make the problem non-trivial.

At the moment the shutter closes, we know almost nothing: a bounding box, maybe a top-1 label from an entity recognizer, some exif metadata, and the child's tier. From that, we must route to one of a growing library of games — and the game must feel bespoke.

Four structural reasons this is hard.

  1. Photos are messy. A plush rabbit might be labeled stuffed_bunny, rabbit, or plush_toy depending on the recognizer's confidence. The matcher today relies on exact string membership against an authored list of canonical neighbors (the constellation map); semantic/embedding fallback is a planned mitigation, not yet implemented. See §09 STRING MATCH.
  2. The entity library is small. Roughly twenty canonical entities have authored tier_guidance under data/mappings_dev20_0318/. Any photograph outside that set has to be bridged to a neighbor before a game can run at all.
  3. The game library is growing. Each pillar adds gold-standard games (one per pillar × category so far) plus property-bridge templates (Detail Detective, Color Scout, Material Lab, …). The matcher must pick the best game from a pool that is wider than the entity pool.
  4. Authors want leverage. A new game written today should reach many entities tomorrow without hand-wiring each pairing. That is why property-bridge templates exist, and why games carry entity_attributes_covered instead of a fixed entity list.
The one-line framing. Matching is not a lookup; it is a routing decision over two asymmetric libraries — entities on one side, games on the other — joined by attribute paths and a constellation of semantic bridges.

Everything in the rest of this document exists to make one of those four problems tractable. §02 describes the three data layers that carry the information. §05 walks the decision path end-to-end. §06 is the one technical rule that ties the pieces together. The rest are examples, checklists, and gotchas.

§ 02 / Three knowledge layers

Entity facts,
neighbor graph, game contracts.

The matching system reads from three independent sources. Each owns a different kind of knowledge and each is maintained by a different cadence. Confusing them is the most common authoring mistake.

01

Entity YAMLs

Per-entity tier_guidance organized as tier_0/1/2 × dimensions (appearance, senses, function, context, emotions…) × attributes. Each attribute carries a literal value string plus child-facing topics. This is the entity's "what we know about it" record.

data/mappings_dev20_0318/<domain>/<file>.yaml
e.g. daily_objects/bath_toys.yaml (rubber_duck)
02

Constellation map

Per-entity routing table for unmapped photographs. Each entry declares mapped_entity, category, pillar, gold_standard, bridge_type, bridge_strength, bridge_prompt_pattern, and a constellation of semantic neighbors. Never bridges across categories.

data/constellation_map.yaml
v0.3 · 20 entries (12 gold + 8 legacy)
03

Activity designs

Each game's §A.1 entity_attributes_covered lists the dotted attribute paths it exercises. Bound games (e.g. teddy_bear_cat1_gold) list real IDs from one entity YAML. Parameterized templates (e.g. detail_detective_property_gold) list abstract paths. Bound games also carry a §A.2 Constellation Adaptation Notes block (Preserve / Swap / Watch).

designs/cat1/*.md · designs/cat5/*.md
per-design §A.1 and §A.2 sections

How the three layers compose

The entity YAML answers "what are the observable and conceptual hooks on this entity?" The constellation map answers "if I don't know this entity, what mapped entity should I pretend it is?" The activity design answers "which hooks does this game actually exercise, and if the entity is only a bridged neighbor, what must I preserve, swap, or watch out for?"

Each layer is authored independently. A new entity YAML does not require editing any game design. A new game design does not require editing any entity YAML. A new constellation entry does not require editing either — it adds a neighbor, and the existing game designs adapt via §A.2. This decoupling is the point.

Rule of thumb. If your change is about what the object is, edit the entity YAML. If it's about what game to run when, edit the design. If it's about which unknown objects should route to a known one, edit constellation_map.yaml. Don't cross layers.

**Preserve** — must not change across neighbors:
- The 3-round escalation (physical → emotional → combined)

**Swap** — re-phrase for the bridged entity:
- "Teddy is shivering" → neighbor's cold-signal (bunny: ears drooping, doll: icy hands)

§ 03 / Core concepts glossary

Twelve terms,
three sentences each.

Every other section reuses these twelve terms exactly as defined here. When in doubt, read this section first.

Entity
The object the child photographed. May be mapped (has its own YAML under data/mappings_dev20_0318/) or unmapped (not in the registry but possibly in the constellation of a mapped entity).
Attribute
A single authored property of an entity inside its tier_guidance block. Keyed as tier_N.<dimension>.<attribute> (e.g. tier_0.appearance.body_color). Each attribute owns a literal value and a set of child-facing topics.
Property
A generic cross-entity dimension like color, shape, texture, material. Property-bridge templates parameterize against properties, not against specific entities. One property can resolve to many attribute paths across the entity library.
Gold design
A pillar-owning, entity-coupled activity design (e.g. teddy_bear_cat1_gold, butterfly_cat5_gold). Twelve exist — one per pillar × in-device / out-of-device category. They are bound games with rich §A.1 attribute coverage and a §A.2 adaptation block.
Property-bridge template
A parameterized game design (e.g. detail_detective_property_gold, color_scout_property_gold). It declares abstract attribute paths under §A.1 and substitutes the matched entity's attribute value into the game at runtime.
entity_binding
The declared coupling strength of a game: bound (one entity, strict overlap), parameterized (property template, loose overlap), or agnostic (runtime-decided, legacy). See §06 for the dual rule. Derived from top-matter by tooling — authors rarely set it by hand.
Constellation
The list of semantic neighbors under a mapped entity's entry in constellation_map.yaml. An unmapped photographed entity is matched by searching these lists. Never crosses category.
bridge_type
The semantic relation a constellation edge encodes: same_taxonomy, shared_habitat, same_role, same_domain, shared_affordance, part_of, or companion. Drives the bridge_prompt_pattern wording the AI uses.
bridge_strength
Ordinal preference — very_strong > strong > moderate > weak. Used to rank candidates when one unmapped entity matches multiple mapped entities' constellations.
§A.1 (entity_attributes_covered)
The required tag-block field inside each activity design that enumerates the attribute paths the game exercises. For bound games it is a concrete list of IDs from one entity YAML; for parameterized templates it is a short list of abstract paths. Consumed by the matcher.
§A.2 (Constellation Adaptation Notes)
Adaptation recipe inside each bound gold design. Three blocks: Preserve (must not change across neighbors), Swap (re-phrase for the bridged entity), Watch (gotchas to avoid). Applied when the matcher picks a bound game for a bridged (non-gold) entity.
Dual-semantics matcher
The rule in program.md §1.9 that reads entity_binding and selects between strict overlap (bound) and loose overlap (parameterized) when scoring entity_attributes_covered against an entity's tier_guidance. See §06 for the full statement.
Category (Cat1 / Cat5)
The structural class of an activity. Cat1 = Sustained Verbal Interaction (in-device, child talks with the AI about the photographed entity). Cat5 = Collection/Tracking Exploration (out-of-device, child moves around the real world collecting evidence). Categories never mix inside the constellation map — a Cat1 unmapped entity only bridges to Cat1 mapped entities.
Tier (T0 / T1 / T2)
The child-age band a game or entity row is tuned for. T0 ≈ ages 2–4 (single-word answers, sensory focus). T1 ≈ ages 4–6 (short sentences, cause/effect). T2 ≈ ages 6–8 (multi-step reasoning, roleplay). Each entity YAML carries tier_0/1/2 sub-blocks; each game declares its Recommended Tier; the matcher penalizes tier mismatch in step 9.
Pillar (Experience Pillar)
Six emotional/cognitive themes every game declares one of: Mystery (deduction, clues), Creation (making/imagining), Performance (showing off, roleplay), Discovery (exploring, hypothesizing), Adventure (quest, journey), Nurture (care, empathy). Drives rank step 9 via child's recent-pillar history.
Recognizer
The vision component that takes the photo pixels and returns a top-1 canonical label plus a confidence score (e.g. {label: "rubber_duck", confidence: 0.91}). This matching guideline treats the recognizer as a black box — its vocabulary and thresholds are owned upstream.
Property detector
A separate vision subsystem that runs only when constellation lookup misses. It returns fixed-schema features: color, shape, material, pattern, each with a value and confidence. Used to route unmapped photos to parameterized templates when no entity neighbor exists.
Registry / _index.yaml
The master index data/mappings_dev20_0318/_index.yaml that lists every canonical entity the system knows, keyed to its YAML file path. Step 3 of the pipeline is a membership test against this file — "is the recognizer's label a known entity?"
Tag block
The canonical metadata record emitted by every activity run. Defined in program.md §1.9. Mandatory fields: entity, activity, pillar, tier, bridge_applied (null / mapped_entity / "property"), plus a progression block with topic_axis and difficulty_level. Downstream consumers (child recap, parent dashboard) only read the tag block — never the raw game script.

§ 05 / Full matching pipeline

Twelve steps
from shutter to game.

One canonical path, with explicit off-ramps. The diagram is a simplified render of docs/diagrams/matching-pipeline-general.excalidraw; the .excalidraw file stays the visual source of truth.

Simplified render · canonical layout in docs/diagrams/matching-pipeline-general.excalidraw
01

Photo in

Child snaps a picture. The matcher receives the image, a tier (T0/T1/T2), and — optionally — exif or last-session context. No text from the child yet.

02

Entity recognizer

Vision model returns a candidate entity label with confidence (e.g. stuffed_bunny · 0.82). The matcher uses this as the initial key. Low-confidence calls are held for a tie-break in step 9.

03

Is the entity mapped?

Membership test against data/mappings_dev20_0318/_index.yaml. If yes, we load that entity's tier_guidance directly (step 4a). If no, the entity is unmapped and we route to the constellation check (step 4b).

04

Mapped path · load tier_guidance

The entity's YAML is loaded in full. Its flat attribute paths (tier_N.dim.attr) are the set the matcher overlays onto each candidate game's §A.1. Proceed to step 7.

05

Unmapped · check constellation_map

Scan data/constellation_map.yaml for an entry whose constellation list contains the unmapped entity's label. On hit, use the corresponding mapped_entity as a proxy: load its tier_guidance and fire the bridge_prompt_pattern so the child hears the semantic bridge ("Your stuffed_bunny is a cozy friend, just like a teddy bear!"). Proceed to step 7.

06

No constellation hit · detect a property

Last resort before the conversation-only off-ramp: try to detect a generic visual property — a color, a shape, a material — from the photo directly. If a property is recoverable, route to the set of parameterized property-bridge templates. The matched property value becomes the runtime substitution for the template's {property} parameter.

07

Build the candidate set

Collect every game whose §A.1 entity_attributes_covered overlaps the entity's attribute paths. Bound games contribute when their full list is a subset; parameterized templates contribute when any one of their abstract paths is a subset. Typical set size is 5–20 entries — because common paths like tier_0.appearance.color appear in almost every mapped entity, most parameterized templates land in the set on every run. Step 09 (rank) is what narrows to one. Empty set is an error — fall back to a conversation-only session. Overlap is exact-string on the full dotted path — body_color ≠ color; see §09 STRING MATCH.

08

Apply the dual-semantics filter

This is the heart of the matcher. For each candidate, dispatch on entity_binding: bound → require strict overlap; parameterized → loose overlap; agnostic → runtime-chosen rule. Candidates failing their rule are dropped. See §06 for the formal statement.

09

Rank survivors

Sort by: bridge_strength (if bridged) → pillar fit (child's recent history vs. candidate pillar) → tier match against target tier → recency penalty (avoid repeating the last game). Ties break toward higher §A.1 coverage count.

10

Pick the top game

One game wins. Its activity design becomes the runtime script. If the entity was bridged via the constellation, the §A.2 block of the chosen bound game is now in scope.

11

Apply adaptation

For a bridged run, read §A.2 Constellation Adaptation Notes: Preserve the mechanics that must stay identical, Swap the scripted references for the actual photographed entity, and Watch the per-neighbor gotchas. For a parameterized template, substitute {property} with the matched value.

12

Emit the tag block · downstream consumers read

The chosen game emits the tag block defined in program.md §1.9 (activity_id, entity, category, pillar, tier, attributes, related_concepts, key_concepts, ATL skills, progression axis, …). The child recap and parent growth-path dashboard read from this block. Matching is done.

Where §A.2 enters. Notice the §A.2 adaptation block is only read at step 11 — after a game is already chosen, and only when the entity reached this pipeline by bridging. A direct mapped-entity run skips §A.2 entirely and uses the design's scripted dialogue verbatim.

Four worked traces · one per path class

Pre-computed walks through the pipeline against real fixture data (data/mappings_dev20_0318/*, data/constellation_map.yaml, and the 12 gold + 18 property-bridge designs under designs/). No live matcher — these are authored traces, not runtime output. Use them as reference runs when authoring new designs or diagnosing why a photographed entity took the path it took.

Quick vocabulary. Each trace uses these terms — all defined in §03 if you need the long version, but here's the one-liner per concept so the traces read straight through:
  • Recognizer — vision model that turns a photo into {label, confidence} (e.g. {"rubber_duck", 0.91}).
  • Registry / _index.yaml — the list of ~20 canonical entities the system knows about.
  • tier_guidance — the authored-facts block inside each entity YAML, flattened to dotted paths like tier_0.appearance.body_color.
  • §A.1 — per-game attribute list: "these are the attribute paths this game exercises."
  • §A.2 — per-gold-game adaptation recipe: Preserve / Swap / Watch, applied when the entity came in via bridging.
  • Constellation map — authored table of neighbors: "if you see X, treat it as mapped entity Y and say Z to introduce the bridge."
  • Property detector — vision subsystem returning {color, shape, material, pattern} with confidences; runs only when constellation lookup misses.
  • Bound game — written for one specific entity (strict overlap). Parameterized template — written for any entity exposing a property (loose overlap). See §06.
  • Tag block — the metadata record the chosen game emits at the end; what downstream recap + parent dashboard consume.

§ 06 / Matcher semantics · the dual rule

Two overlap rules,
one dispatch.

This is the only formal rule in the matching pipeline. Everything else is plumbing. The normative statement lives in program.md §1.9 "Matcher semantics"; this section is an explanatory mirror of that spec.

Each activity design declares an entity_binding. The matcher dispatches on that field to decide how to compare the game's §A.1 entity_attributes_covered against the entity's tier_guidance attribute paths.

entity_binding: bound

Strict overlap

The game is written for one specific entity. Every ID in entity_attributes_covered must resolve to an attribute: entry under that entity's tier_guidance. Missing IDs are lint errors, not soft mismatches.

§A.1 ⊆ entity.tier_guidance.paths
entity_binding: parameterized

Loose overlap

The game is a property-bridge template. Its §A.1 lists 2–4 abstract paths that any qualifying entity could expose. An entity qualifies if at least one path is present in its tier_guidance; the matched attribute's value fills {property} at runtime.

§A.1 ∩ entity.tier_guidance.paths ≠ ∅
entity_binding: agnostic

Runtime-decided

Rare, mostly legacy. The field is still required for completeness, but the matcher may apply either strict or loose overlap (or a category-specific one) based on the design's declared style and pillar at runtime.

dispatch(style, pillar) → {strict | loose | custom}

Why two rules

A bound game is written against a single entity's actual hooks — Teddy's Care Station knows teddy is plush and might be shivering because tier_0.senses.touch_feel and tier_0.emotions.comfort_signal exist in that entity's YAML. The game would not make sense if any of those hooks were missing. Strict overlap enforces that.

A parameterized template is the opposite. Detail Detective works for any entity with a visible property — color, shape, pattern, texture — because its mechanic is the game loop, not the object. It cannot require all four paths; requiring even two would shrink the reachable entity pool by half. Loose overlap keeps the template's scope wide while still guaranteeing the matched property has an authored value string to substitute at runtime.

Inference from top-matter

Authors do not set entity_binding by hand. Tooling derives it from the design's header: Trigger Entity: {entity_name} → bound; Trigger Entity: Any entity with a detected {property}… plus the word "Property Bridge" → parameterized. Authors only have to pick the overlap semantics implicitly, by choosing which attribute IDs to list.

Normative pointer

Source of truth. program.md §1.9, subsection "Matcher semantics". If this document and program.md ever disagree, program.md wins and this section is out of date — file a PR to re-sync. The pre-output self-check for §A.1 also lives in program.md §1.9 and is mirrored in §08 below.

§ 08 / Authoring checklist

Three recipes,
one discipline.

Three things you might be adding: a bound game, a parameterized template, or a new entity YAML. Each has a short list of must-dos that keep §05 working. Mirror of the pre-output self-check in program.md §1.9.

Recipe 1 · Adding a bound game

Entity-coupled gold design (e.g. lion_cat1_gold, flower_cat5_gold).

  • Pick a mapped entity. Confirm it has a tier_guidance block in data/mappings_dev20_0318/ and an entry in _index.yaml.
  • Populate §A.1 entity_attributes_covered with at least 4 attribute IDs drawn from that entity's tier_guidance. Every ID must resolve — run the lint step below.
  • Spread the IDs across tier_0, tier_1, and (where the game supports it) tier_2. A bound game with all IDs at tier_0 will be un-rankable for older children.
  • Write the §A.2 Constellation Adaptation Notes block: Preserve (mechanics that must stay identical), Swap (per-neighbor vocabulary substitutions), Watch (per-neighbor gotchas). Check the constellation entry in constellation_map.yaml for the neighbor list.
  • Confirm the dual-semantics outcome: the game is bound, strict overlap must pass against the target entity.
  • Run the program.md §1.9 pre-output self-check. All required tag-block fields, no placeholders, all IDs resolve, all self-check boxes tick.

Recipe 2 · Adding a parameterized property template

Game that runs on many entities by reading one property (e.g. detail_detective, material_lab, shape_quest).

  • Pick a property category (color, shape, pattern, texture, material, size, sound, …). The template's entire mechanic has to flow from that one property.
  • Populate §A.1 with 2–4 abstract attribute paths that real entities are likely to expose. Pick paths that appear in ≥3 entity YAMLs — one-entity templates are effectively bound games and should be re-authored as bound.
  • Keep §A.1 short. Loose overlap means any one hit qualifies; a long §A.1 makes the template look heavier than it is at ranking time.
  • Use the {property} substitution convention throughout the design. The runtime will inject the matched attribute's value string — if the script hardcodes "red" or "wheels" it will break for non-truck entities.
  • Confirm the dual-semantics outcome: entity_binding: parameterized, loose overlap, and the pre-output self-check passes (see program.md §1.9).
  • Do not write a §A.2 block for parameterized templates — the adaptation happens at the parameter level, not at the neighbor level.

Recipe 3 · Adding a new entity YAML

New mapped entity (e.g. adding frog under animals/amphibians.yaml).

  • Pick the right domain folder under data/mappings_dev20_0318/ (animals/, daily_objects/, plants/, natural_phenomena/, …). Match the shape of the sibling YAML files exactly — downstream readers key on literal field names.
  • Populate tier_guidance with tier_0, tier_1, and optionally tier_2 dimensions. Each attribute needs a value string and 2–3 child-facing topics.
  • Add the entity to _index.yaml. Without this the registry check at step 3 will miss it.
  • Pick a neighbor mapped entity in constellation_map.yaml whose category and pillar fit, and add the new entity to that neighbor's constellation list — or write a new top-level constellation entry if the new entity is itself a gold standard.
  • Verify that at least one existing bound game's §A.1 overlaps the new entity's tier_guidance or at least one parameterized template's §A.1 matches. If neither, the matcher will fall back to conversation-only — that is an authoring gap, not the matcher's fault.
  • Sanity-check: pick two random attribute IDs from the new YAML and confirm the game you expect to route to actually lists them (bound) or can loosely match them (parameterized).
§ 09 / Edge cases and gotchas

What breaks,
and what to do.

The pipeline has explicit off-ramps; the authoring layer has a few silent ways to drift. These are the ones to watch.

STRING MATCH, NOT SEMANTIC

Steps 5–7 compare strings, not meanings

The whole matcher is exact-string comparison on canonical labels and dotted-path keys. Constellation membership is a literal recognizer_label in entry.constellation[*] test; candidate-set overlap is set intersection on the flattened path strings. If the recognizer returns plush_rabbit but the list has stuffed_bunny, it's a miss — fall-through to property detection. If one entity YAML uses tier_0.appearance.body_color and another uses tier_0.appearance.color, games bound to one can't reach the other unless the game's §A.1 enumerates both paths. Known drift in this repo: rubber_duck uses body_color, teddy_bear uses color; property templates standardize on color. Future mitigations (none implemented yet): an authoring vocabulary spec for attribute suffixes, recognizer label canonicalization (snake_case, lemmatize), a synonyms: field in constellation_map entries, an embedding-based nearest- neighbor fallback, or an LLM fallback before the off-ramp. For now: author discipline is the only safeguard.

UNMAPPED & UN-BRIDGED

Entity in no constellation

Photo recognized as an entity that appears in no constellation list and exposes no detectable property. Pipeline takes the conversation-only off-ramp. Fix: add the entity to a sibling constellation, or accept the fallback until the entity earns a gold design.

EMPTY OVERLAP

Game with zero §A.1 hits

A bound game's §A.1 lists IDs that no longer exist in the entity YAML (e.g. after a tier_guidance refactor). Strict overlap fails silently; the game drops out of the candidate set. Fix: run the §1.9 pre-output self-check lint — unresolved IDs are errors, not warnings.

TIES AT RANK

Multiple games survive equally

Two bound games for the same entity, identical bridge_strength, equal §A.1 coverage. Current tiebreaker: highest entity_attributes_covered count, then alphabetical by activity_id. Author action: differentiate by pillar so the ranker can lean on curiosity-profile history.

LOW CONFIDENCE

Recognizer returned ambiguous label

Entity label at 0.4 confidence. Matcher can treat the label as probabilistic and expand the candidate set across top-N labels, or defer to parameterized templates only (safer). Default: when in doubt, property-bridge — it degrades gracefully.

PATH TYPOS

§A.1 ID that does not resolve

Typical: tier_0.apperance.color (missing a), tier_0.sense.squeak_sound (singular). Strict overlap fails for bound games; loose overlap silently under-matches for templates. Fix: lint §A.1 against the live YAML at CI time.

TIER MISMATCH

Game supports only T0, child is T2

The ranker penalizes tier mismatch. If no game at any tier matches the entity, the fallback is the conversation-only off-ramp — never stretch a T0 game to a T2 child. Author action: either widen the game's tier_variants or ship a tier-T2 sibling design on the same axis.

CROSS-CATEGORY BRIDGE

Constellation entry that spans Cat1 ↔ Cat5

Forbidden by the constellation-map contract ("Never bridge across categories"). A Cat5 unmapped entity must only appear in a Cat5 mapped entity's constellation. Lint rule: verify consistency at _index.yaml load time.

§A.2 DRIFT

§A.2 references a neighbor no longer in the constellation

Design's Swap block mentions sock_monkey; constellation_map.yaml removed it in a cleanup pass. Bridged runs for the removed neighbor now use §A.2 verbatim without validation. Fix: CI check that §A.2 Swap entries ⊆ the mapped entity's current constellation list.

§ 10 / Cheat sheet

One page,
two lenses.

Read the matcher from the runtime side (left) or from the authoring side (right). Both lenses are the same rule set viewed differently.

Finding the game from a photo

Runtime · 10 second mental model
  1. Is the entity mapped? Check _index.yaml. If yes, load its tier_guidance.
  2. If not, is it in a constellation? Scan constellation_map.yaml for a hit. Use the mapped neighbor as a proxy.
  3. If still not, is there a detectable property? If yes, route to parameterized templates. If no, fall back to conversation-only.
  4. Build the candidate set. Every game whose §A.1 overlaps the entity's (or proxy's) tier_guidance.
  5. Apply the dual filter. bound → strict; parameterized → loose.
  6. Rank. bridge_strength · pillar fit · tier match · §A.1 coverage.
  7. Adapt. Bridged runs read §A.2 (Preserve / Swap / Watch). Parameterized runs substitute {property}.
  8. Emit tag block. Downstream surfaces (child_recap, parent dashboard) read from it.

Writing content that will be findable

Authoring · 10 rules that keep you matchable
  1. Bound games: list 4+ real IDs in §A.1. Every one must resolve. Spread across tiers.
  2. Parameterized templates: list 2–4 abstract paths. Every path must appear in ≥3 entity YAMLs.
  3. Never cross bind-types: a property template never names a specific entity; a bound game never uses {property} substitution.
  4. §A.2 is only for bound games. Property templates adapt via parameters, not neighbors.
  5. New entity YAML: add to _index.yaml AND to a constellation, or it is effectively invisible to unmapped-lookup.
  6. Constellations never cross category. Cat1 entity ↔ Cat1 neighbors only.
  7. Field names matter. Downstream keys on literal strings — tier_guidance, dimensions, attribute, value, bridge_prompt_pattern, constellation.
  8. Run the §1.9 self-check. If any box fails, the tag block is a silent runtime bug.
  9. If your change is about the object, edit the entity YAML. If it's about the game, edit the design. If it's about routing, edit the constellation map.
  10. Normative source is program.md §1.9. This document is the explanatory mirror — if they disagree, §1.9 wins.
